最近大家都在讨论：

MCP 已死，Skill + CLI 才是王道。

本质上，大家讨论的是 **谁来充当 AI Agent 的外部能力**？

在这里先说一个结论： **根据不同的场景，MCP 和 Skill + CLI 各有用处**。

以下，仅代表我的个人观点，如有考虑欠缺，欢迎讨论并补充。





# 1. 关于 CLI

CLI（Command-Line Interface）：**命令行界面**。



* **CLI 是如何工作的？**

  用户（用户可能是人、可能是 agent ）在终端里输入一条命令，然后电脑就会去做一件事。

  比如：`mysql -u root -p`

  意思是，让电脑去打开 MySQL 数据库。

**CLI 本质上**是：用户在命令行里输入一段命令，操作系统找到对应的可执行程序，然后把这段命令里的参数交给这个程序执行，于是程序去完成某件事。

整个过程看起来像是：“用户在命令行输入一段命令，让电脑去干某些事情”



* CLI 命令 对应的可执行程序的运行（**CLI 命令的正常运行**），**非常依赖于个人本机电脑的环境配置**

  比如说，在终端输入 `pnpm dev`，电脑并不是天然就认识这条命令，这条命令的正常执行，需要你的本机电脑中**安装 `pnpm`这个程序，这个程序的运行还要依赖 `Node.js`、项目依赖等等**，非常**依赖个人本机电脑的环境配置。**



# 2. MCP

* **MCP 是什么？**

  MCP（Model Context Protocol）：模型上下文协议。

  不过，我们这里讨论的是 **MCP Server**：一个按**照 MCP 这套协议**，对外提供工具和数据能力的 **服务端程序**



* **MCP Server 是如何工作的**？

  MCP Server 本质上是 把能力封装成 AI 可发现、可调用、可读取的标准化接口。






# 3. MCP 使用场景



由于 Skill + CLI，CLI 的运行非常依赖个人本机电脑的环境，如果一个大型团队想统一 AI Agent 的外部能力，最好的方式就是使用 MCP，只要外部系统（比如 数据库服务）实现MCP，这样就**无需考虑团队中每个人的本机电脑环境是怎样**，就可以直接让 AI Agent 连接这个 MCP Server。

以相对小的代价统一了团队 AI Agent 的外部能力。

> 对大型团队来说，如果想统一 AI Agent 的外部能力，MCP 往往比 skill + CLI 更适合。因为 skill + CLI 更像把能力分发到每个人本机，而 MCP 更像集中提供一个统一服务入口。这样不是完全没有环境问题，而是把环境和依赖的管理，从“每个人各自维护”转成“团队集中维护 MCP Server”。





# 4. Skill + CLI 使用场景

对于个人开发者，或者团队里想快速试验自己想法的人，更适合优先使用 Skill + CLI

你可以使用 Skill + CLI。

* **为什么呢**？

  * **AI Agent 接入 MCP Server 的缺点**

    单个 github MCP Server ，光是agent加载这个 mcp server 就大约消耗 30K token 的上下文空间。
    所以如果你连接了 飞书 mcp、某个数据库的mcp 等多个系统的mcp，光是加载可能就消耗数十万token的上下文空间。
    有时候，仅仅是 mcp 的工具说明（JSON Schema），可能就占用了 可用上下文的 40% 到 60%。

  * 因为 **Skill + CLI 更轻、更快、更适合本机工作流**。
     Skill 负责告诉 AI“这件事应该怎么做”，CLI 负责让 AI 真正去执行命令；而 Skills 本身也是按需加载的，`SKILL.md` 是核心说明，其它参考文件和脚本只在需要时再加载。相比之下，MCP 更偏向“把外部能力做成标准化接口”，适合统一接入，但也可能带来额外的上下文和集成成本。

* 使用 Skill + CLI 的主要优点

  1. **上下文更省**

     对于很多 coding agent 场景，CLI 往往比 MCP 更省上下文。比如：Playwright 官方已经明确把 `playwright-cli` 定位为更适合 coding agents 的方案，因为它能避免加载大型工具 schema 和冗长的页面结构数据。

  2. **更适合本地开发和快速试验**

     个人开发者最常做的事通常是：跑测试、看日志、构建项目、执行脚本、查本地数据库、处理文件。这些事情本来就天然适合 CLI，不需要先搭一个正式的服务层。**运行一个 CLI 命令，就能执行跑测试等等**。Claude Code 的 skills 机制也支持把脚本、模板、参考文档配合 `SKILL.md` 一起使用，很适合这种“先把事情做成”的工作流。

  3. **你可以完全按自己的想法定制流程**

     如果你是个人开发者，或者团队里想自己试一套新方法的人，你可以自由决定：

     - 先查 schema 还是先跑查询
     - 用什么命令
     - 输出什么格式
     - 怎样做 dry-run
     - 怎样限制只读
        这种灵活性对个人试验特别有价值。CLI 本质上就是执行程序，规则可以由你自己定义。

> 对于个人开发者，或者团队里想快速验证自己想法的人，Skill + CLI 往往是更合适的起点。因为它更轻、更灵活、更贴近本地开发工作流，也通常更省上下文。MCP 更适合把外部能力做成标准化、可共享、可治理的接口；但在个人场景下，如果过早接入很多 MCP server，工具 schema、资源描述和多系统集成可能会带来明显的上下文与复杂度成本。
