如何优化 skill 的 `description` 字段，使它在用户提出相关请求时，能够更准确、更稳定地触发。

**一个 skill 只有在被激活时才真正有用**。你在 `SKILL.md` 的 frontmatter 中填写的 `description` 字段，是 agent 判断在某个任务下是否应该加载这个 skill 的主要依据。描述写得不够具体，skill 就可能在该触发的时候没有触发；描述写得过于宽泛，skill 又可能会在不该触发的时候被误触发。

本指南介绍如何**系统地测试**并改进你的 skill 描述，以提升触发准确性。

# 1. skill 是如何被触发的

agent 一开始不会把所有 skill 的完整内容都读进去，不然上下文会很快塞满（**渐进式披露**）。它先只看每个 skill 的“ name 字段（名字） + description 字段（简介）”，像先看目录一样。等发现**某个用户请求**和这个**简介对上了**，它才会**进一步加载完整的 `SKILL.md`**（**注意，是只加载完整的 `SKILL.md`，其他的`/script`等文件夹的内容，是在`SKILL.md`文件中指定触发的**），再按里面的规则工作。

这意味着，**`description` 字段承担了触发 skill 的全部责任**。要是这个描述没有说明清楚这个 skill 在什么情况下有用，agent 就不知道该在什么时候调用它。

还有一个很重要的细节：agent 通常只会在任务需要**超出它自身基础能力**的知识或能力时，**才去查找和调用 skill。**像“读一下这个 PDF”这种简单、单步骤的请求，**即使 description 写得再匹配**，**也未必会触发 PDF skill**，**因为 agent 用基础工具就能完成**。

真正容易体现 skill 描述价值的，通常是那些**需要专门知识的任务**——比如不熟悉的 API、某个领域特有的工作流，或者不常见的格式。在这些场景下，一个写得好的 description，往往就会**直接影响 agent 能不能正确**想到并调用这个 skill。



# 2. 编写有效的 description 字段

在开始测试之前，先了解一个好的 description 应该长什么样，会很有帮助。这里有几个原则：

* description 最好不要只是介绍“这个 skill 是干什么的”，而要**直接告诉 agent“什么时候该用它”**。所以要写成指令式，比如**“当遇到……任务时，使用这个 skill”。**
* **你要写的是“用户想做什么”**，而不是“这个 skill 底层怎么实现”。因为 agent 看的不是技术原理，而是用户的请求是不是和这个 skill 对得上。
* description 不要写得太保守。**凡是这个 skill 可能派上用场的场景**，**都尽量直接写出来**。哪怕用户没有把关键词说出来，**只要实际意图对得上**，也要让 agent 能想到这个 skill。
* description **不要写得太长**。一般几句话就够了：既能把用途说清楚，又不会占太多上下文空间。而且规则里已经明确限制了，最长不能超过 1024 个字符。



# 3. 设计用于测试 skill 触发效果的查询语句

为了测试 skill 的触发情况，你需要一组 eval queries：这些都是**尽量贴近真实用户表达的提示语**，并且你要事先标明每一条“应该触发”还是“不应该触发”你的 skill。

```txt
[
  {
    "query": "我在 ~/data/q4_results.xlsx 有一个电子表格，C 列是收入，D 列是支出——你能帮我新增一列利润率，并把低于 10% 的内容高亮出来吗？",
    "should_trigger": true
  },
  {
    "query": "把这个 json 文件转成 yaml，最快的方法是什么",
    "should_trigger": false
  }
]
```

建议准备大约 20 条查询：其中 8 到 10 条应该触发，另外 8 到 10 条不应该触发。

## 1. 应该触发 skill 的查询

这些查询是用来检验 description **有没有**把 skill 的范围**覆盖到位**。设计时，可以沿着几个不同维度来做变化：

* **表达方式**：有些写得正式一些，有些更口语化一些，还有一些可以故意带上拼写错误或缩写。
* **明确程度**：有些查询会直接点出这个 skill 所属的领域（例如 “analyze this CSV” / “分析这个 CSV”），而另一些则不会直接说出领域名称，只是描述自己的需求（例如 “我老板想让我用这个数据文件做一张图表”）。
* **信息详细程度**：测试时要同时包含简洁型提示和上下文丰富型提示。比如，既有“分析我的销售 CSV 并生成图表”这种短句，也有那种附带文件路径、字段名和任务背景的长消息。（**有的人一句话就说完需求，有的人会讲一大段背景。所以测试时，短的和长的都要有**。）
* **任务复杂度**：不要全是简单题。要有些请求很直接，一步就做完；也要有些请求很复杂，里面夹着很多步骤。这样才能看出来：即使这个 skill 只是整个任务中的一小部分，agent 还能不能想到要用它。

最能测出 description 水平的查询，不是那种“一眼就该触发”的题，而是**那种“其实该用这个 skill，但表面上看不太明显”的题**。因为简单题大家都会，**难的是这种不那么直接的情况**。



## 2. 不应该触发 skill 的查询

**最值得重视的“不应触发”测试样本**，不是完全不相关的题，而是**那种“看起来有点像、其实不该触发”的题**。它们能帮你看出 description 是不是真的写得准，而不是只会靠关键词乱匹配。

如果以一个 CSV 分析 skill 为例，那么下面这些“不应触发”的测试样本，**并不算特别有测试价值**：

* “**写一个斐波那契函数**”——这和 CSV 分析显然毫无关系，因此几乎没有测试价值。
* “**今天天气怎么样**？”——连相关关键词都没有，太容易判断为不匹配了。



**更有价值的负向例子**：

* “**我需要修改 Excel 预算表中的公式**”——虽然它和 skill 在“表格”“数据处理”这些概念上有重叠，但用户真正需要的是 Excel 编辑，不是 CSV 分析。
* “**你能帮我写个 Python 脚本，把一个 csv 读进来后逐行上传到我们的 Postgres 数据库吗**？”——虽然提到了 CSV，但这个任务的重点是数据库导入/ETL，不属于 CSV 分析。



## 3. 让测试样本更贴近真实情况的技巧

真实用户发出的请求，通常会包含很多普通测试语句里没有的上下文。所以在设计查询时，可以加入这些元素：

* **文件路径**（例如 `~/Downloads/report_final_v2.xlsx`）
* **个人背景信息**（例如“我经理让我……”）
* **具体细节**（例如列名、公司名、数据数值）
* 口语化表达、缩写，以及偶尔出现的拼写错误



# 4. 如何测试一个 description 能否成功触发

最基础的做法是：先把 skill 安装到 agent 里，然后**逐条输入这些查询**，看看 agent 会不会触发并调用这个 skill。同时还要确认，这个 skill 对 agent 来说是**已经注册好且可被发现的**。至于具体注册方式，不同客户端会不一样，比如可能是放在某个 skills 目录下、写进配置文件里，或者通过命令行参数启用。

多数 agent 客户端都会提供一些便于排查的观察手段，例如运行日志、工具调用记录，或者 verbose 输出，你可以借此确认 agent 在执行过程中到底有没有读取你的 skill。一般 agent 都**会留下一些“运行痕迹”**，你可以从日志或**详细输出里看它有没有读你的 skill**。判断标准很简单：**读了 `SKILL.md`，就算触发；没读，直接干活了，就没触发。**

一条查询在以下情况下算“通过”（passes）：

* `should_trigger` 为 `true`，并且这个 skill 确实被调用了；
* `should_trigger` 为 `false`，并且这个 skill 确实没有被调用。



## 1. 多次运行测试

同一道题，agent 不一定次次反应都一样。所以不能只测一次。最好每条题多跑几遍，比如跑 3 次，然后看看它有几次真的触发了 skill，这个比例就是触发率。

判断是否通过时，可以设定一个触发率阈值，默认用 `0.5` 通常比较合适：

* **应当触发的查询**：触发率 **高于** 这个阈值，算通过；
* **不应触发的查询**：触发率 **低于** 这个阈值，算通过。

假设你有 20 个查询，每个都跑 3 遍，那一共就是 60 次调用。这个数量已经很适合用脚本自动化了（手动做会很麻烦，所以最好写脚本）。下面这段只是一个模板，你需要把里面调用 agent 的方式、还有“怎么判断 skill 有没有触发”的部分，改成你自己工具对应的写法。

`!/bin/bash`：

```txt
#!/bin/bash
QUERIES_FILE="${1:?Usage: $0 <queries.json>}"
SKILL_NAME="my-skill"
RUNS=3

# 这个示例是基于 Claude Code 的 JSON 输出，判断运行过程中是否调用了 Skill 工具。
# 你需要把这个函数替换成适用于你所使用 agent 客户端的检测逻辑。
# 如果 skill 被调用了，就应返回 0（成功）；否则返回 1。
check_triggered() {
  local query="$1"
  claude -p "$query" --output-format json 2>/dev/null \
    | jq -e --arg skill "$SKILL_NAME" \
      'any(.messages[].content[]; .type == "tool_use" and .name == "Skill" and .input.skill == $skill)' \
      > /dev/null 2>&1
}

count=$(jq length "$QUERIES_FILE")
for i in $(seq 0 $((count - 1))); do
  query=$(jq -r ".[$i].query" "$QUERIES_FILE")
  should_trigger=$(jq -r ".[$i].should_trigger" "$QUERIES_FILE")
  triggers=0

  for run in $(seq 1 $RUNS); do
    check_triggered "$query" && triggers=$((triggers + 1))
  done

  jq -n \
    --arg query "$query" \
    --argjson should_trigger "$should_trigger" \
    --argjson triggers "$triggers" \
    --argjson runs "$RUNS" \
    '{query: $query, should_trigger: $should_trigger, triggers: $triggers, runs: $runs, trigger_rate: ($triggers / $runs)}'
done | jq -s '.'
```

> 如果你的 agent 客户端具备这个能力，那么当结果已经可以判断时，就没必要把整次运行执行到底：例如，agent 已经读取了 skill，或者它已经在未读取 skill 的前提下开始干活。提前结束这些运行，可以明显减少完整 eval 集的测试时间和开销。（**我们的目的，只是为了测试，是否成功触发这个 skill，没必要消耗更多的 Token 去执行这个 skill，当已经得到我们想要的结果时，直接中断它即可。**）



# 5. 如何用 train/validation 划分避免过拟合

如果你总是对着同一批题去改 description，最后很可能变成“只会做这套题”。看起来测试分数很高，但用户换一种说法，它就可能触发不准了。

一个解决方案是：将这组查询**划分开使用**：

* **训练集（大约占 60%）**：这部分查询用来发现触发失败的情况，并作为优化 description 的依据。
* **验证集（大约占 40%）**：这部分查询先不参与优化过程，只在之后用来验证你的改动是否对未见过的表达方式同样有效。

训练集和验证集都应该按大致相同的比例，包含 should-trigger 和 should-not-trigger 两类查询。不要把所有应该触发的样本都集中到某一边。可以先随机打乱数据，然后把这次划分固定下来，后面每轮测试都用同样的划分方式，这样才能保证前后结果可比。

如果你是用前面那种脚本来做测试，可以直接把查询分成两个文件保存：一个是 `train_queries.json`，一个是 `validation_queries.json`，之后分别拿它们跑脚本即可。



# 6. 反复优化的循环过程

1. 先分别在 train set 和 validation set 上测试当前的 description。训练集结果帮助你发现问题并决定怎么改，验证集结果则用来检验这些改动是否不仅对训练样本有效，也同样适用于新的样本。
2. **在训练集中定位失败样本**：看看有哪些 should-trigger 查询没有成功触发，以及有哪些 should-not-trigger 查询反而被误触发了：
   1. 后续改 description 时，只能参考训练集里的失败样本。无论是你亲自修改，还是通过提示让 LLM 来优化，都应把验证集结果排除在外。（改 description 的时候，只看训练集里做错的题，不要偷看验证集。不然验证集就失去意义了。）
3. 把这段描述再改一改，别只对当前情况有效，要尽量让它在更多类似场景里都能适用：
   * 如果“该触发的没触发”，那大概率就是你把这段说明写得太死、太局限了。要么把范围放大一点，要么把“什么时候该用这个技能”说得更清楚。
   * 如果 should-not-trigger 查询被错误触发了，说明这段描述可能写得太宽泛了。你需要补充更明确的限定，说明这个技能**不负责什么**，或者进一步划清它与相邻能力之间的边界。
   * 不要因为某些查询失败了，就把它们的具体关键词硬塞进描述里，这样其实是在过拟合。更应该做的是，识别这些查询背后对应的通用类别或核心概念，然后从那个层面来调整描述。
   * 如果你已经尝试了好几轮优化还是没有明显进展，那就不要再继续做局部微调了。可以试试用一种结构完全不同的方式来写这段描述，因为换个表达框架或句式，有时比反复润色更有效。
   * **检查一下描述是否仍然控制在 1024 个字符以内**——在优化过程中，描述内容很容易越写越长。
4. 不断重复第 1 到第 3 步，直到训练集里的查询全部通过，或者继续优化也已经没有什么实质性提升。
5. 用验证集的通过率来判断哪一次迭代最好，也就是看验证集中有多少查询成功通过。需要注意的是，**最佳描述未必是最后一次产出的版本**；有些前面的版本，虽然没那么“贴训练集”，但在验证集上的表现可能更好，说明它泛化得更稳。

**通常迭代五轮左右就够了。**如果效果一直没有提升，问题可能不在描述本身，而是在查询集上——比如查询太简单、太难，或者标注得不够准确。

> [`skill-creator`](https://github.com/anthropics/skills/tree/main/skills/skill-creator)  这个 Skill 能把这套优化流程一条龙跑完：自动拆分评估集、并行测试触发率、调用 Claude 给出描述改进方案，并生成一个实时更新的 HTML 报告，方便你边跑边看结果。



# 7. 把最终结果用起来

当你选出最佳描述之后：

1. **更新你 `SKILL.md` frontmatter 里的 `description` 字段。**
2. **确认这段描述没有超过 1024 个字符限制。**
3. **再确认这段描述能够按预期触发。**可以先手动试几个 prompt，做一个快速的合理性检查。想测得更严格一些的话，可以另外写 5 到 10 条全新的查询（同时包含 should-trigger 和 should-not-trigger 两类），再用 eval 脚本跑一遍——因为这些查询从来没有参与过优化过程，所以它们能更真实地检验这段描述是否具备泛化能力。

Before and after:

```TXT
# Before
description: 处理 CSV 文件。

# After
description: >
  分析 CSV 及其他表格数据文件，包括计算统计摘要、添加衍生列、
  绘制图表，以及清理格式混乱的数据。当用户拿着 CSV、TSV
  或 Excel 文件，希望查看、整理、转换或可视化数据时，
  就可以使用这个技能——哪怕他们没有直接说“CSV”或“数据分析”。
```

改进后的描述，一方面更具体地说明了这个技能能做什么（比如统计汇总、添加派生列、生成图表、清洗数据）；另一方面也更全面地说明了它适用于哪些场景（不仅限于 CSV，还包括 TSV、Excel，而且即使用户没有明确说出相关关键词，也同样适用）。

# 8. 下一步

当你的 skill 已经能够稳定触发之后，下一步就要评估它产出的结果是否足够好。关于如何设计测试用例、给结果打分，以及基于结果继续迭代，可以参考 [Evaluating skill output quality](https://agentskills.io/skill-creation/evaluating-skills)。
