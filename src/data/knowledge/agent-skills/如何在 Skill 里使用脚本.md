如何在你的技能中运行命令，并打包可执行 script。

技能不仅可以让 agent 执行 shell 命令，还可以把能重复使用的脚本放进 `scripts/` 文件夹里。这篇指南主要讲三件事：

1. **只运行一次的命令怎么写**
2. **可以独立运行、自己带依赖的脚本怎么写**
3. **怎么把脚本设计得更适合 AI agent 去调用**



# 1. 只执行一次的命令

当现有的软件包已经能完成你需要的事情时，你可以直接在 `SKILL.md` 的说明里引用它，而不一定非要再建一个 `scripts/` 目录。很多技术生态都提供了工具，可以在运行时自动解析并安装依赖。

`npx` 可以直接帮你执行一个 npm 包，如果你本地没有，它会先下载再运行。`npx` 是 npm 自带的，而 npm 一般在安装 Node.js 时就会一起装上。

```bash
npx eslint@9 --fix .
npx create-vite@6 my-app
```

* 随 Node.js 一起提供——不需要额外安装。
* 它会下载这个包，运行它，并把它缓存起来，方便下次继续使用。
* 为了保证结果可复现，可以用 `npx package@version` 这种方式固定版本。

在技能中使用一次性命令的小技巧：

* **固定版本**（比如 `npx eslint@9.0.0`），这样这个命令在以后执行时也会保持相同的行为。
* 你要把使用这个技能前需要满足的条件，明确写在 `SKILL.md` 里，比如“必须有 Node.js 18+”。不要想当然地觉得 agent 运行的环境里一定已经装好了这些东西。要是属于运行环境本身的要求，就写到 frontmatter 的 `compatibility` 字段里。
* **复杂命令不要一直直接写在说明里，最好单独写成脚本。**
  如果只是简单调用某个工具，再加几个参数，那用一次性命令就够了。
  但如果命令越来越长、越来越复杂，已经很容易写错，那就更适合放进 `scripts/` 里，做成一个测试过的脚本，这样更稳。

# 2. 在 `SKILL.md` 中引用脚本

如果某个文件是放在这个 Skill 文件夹里的，你就**从这个 Skill 的根目录开始写路径**，不用写电脑上的完整路径。AI 代理会自动帮你处理。

在 `SKILL.md` 中列出可用的脚本，这样 agent 才知道它们存在：

`SKILL.md`：

```txt
## 可用脚本

- scripts/validate.sh —— 用于校验配置文件
- scripts/process.py —— 用于处理输入数据
```

然后再指示 agent 去运行它们（`SKILL.md`）：

```txt
## 工作流程

1. 运行校验脚本：
   ```bash
   bash scripts/validate.sh "$INPUT_FILE"
   ```
2. 处理结果：
   ```bash
   python3 scripts/process.py --input results.json
   ```
```

> 不只是 `SKILL.md`，**像 `references/*.md` 这种配套说明文件里，路径写法也是一样的。**你在代码块里写运行脚本的命令时，也要从技能根目录开始写相对路径，因为 AI agent 就是在那个目录下执行命令的。



# 3. 可以独立运行的脚本

如果某段逻辑以后还要反复用，就把它写成一个放在 `scripts/` 里的脚本，并且把它依赖什么直接写在脚本里。这样 AI agent 执行时，只要跑一条命令就行，不用再额外准备依赖文件，也不用先手动安装。

有些编程语言可以直接在代码里写明依赖：

**Python**：

PEP 723 规定了一种标准写法，让你可以直接在脚本文件里写元数据。依赖项要写在 `# ///` 这类标记包起来的 TOML 块中：

`scripts/extract.py`:

```txt
# /// script
# dependencies = [
#   "beautifulsoup4",
# ]
# ///

from bs4 import BeautifulSoup

html = '<html><body><h1>Welcome</h1><p class="info">This is a test.</p></body></html>'
print(BeautifulSoup(html, "html.parser").select_one("p.info").get_text())
```

Run with [uv](https://docs.astral.sh/uv/) (recommended):

```BASH
uv run scripts/extract.py
```

`uv run` 会先临时创建一个独立环境，把脚本里写好的依赖装进去，再执行这个脚本。`pipx` 也可以这样做，比如 `pipx run scripts/extract.py`，而且它同样支持 PEP 723：

* 使用 PEP 508 版本说明符来固定版本，例如：`"beautifulsoup4>=4.12,<5"`。
* 使用 `requires-python` 来限制 Python 版本。
* 使用 `uv lock --script` 生成锁文件，以实现完全可复现。



# 4. 如何把脚本设计得更适合 AI agent 使用

AI agent 在执行脚本时，会通过看 `stdout` 和 `stderr` 的内容，来决定接下来要采取什么操作。所以，只要在脚本设计上做好几个关键点，就能让代理更顺畅地使用这个脚本。



## 1. **避免交互式提示**

> 不要让脚本在运行时弹出提问，等人来输入

这不是建议，而是 agent 运行环境的**强制要求**。AI 代理使用的是非交互式命令行环境，所以它没办法像人一样去回答终端里的提问、输入密码，或者在确认菜单里做选择。**只要一个脚本执行过程中停下来等这些输入，它就会一直挂在那里，没法继续。**

所有需要传给脚本的信息，都要通过命令行参数、环境变量，或者 `stdin` 输入进去：

```txt
# 不好的写法：会一直等输入，导致卡住
$ python scripts/deploy.py
目标环境：_

# 好的写法：直接报错，并告诉你怎么改
$ python scripts/deploy.py
错误：必须传入 --env 参数。可选值：development、staging、production。
用法：python scripts/deploy.py --env staging --tag v1.2.3
```

**不要让脚本跑到一半停下来问用户。 如果缺少参数，应该直接报错，并明确告诉用户怎么正确使用。**



## 2. 给脚本加上 `--help` 参数说明用法

**AI agent 主要是通过 `--help` 的输出内容，来知道你的脚本该怎么用**。所以你要在里面写清楚：这个脚本是做什么的、有哪些参数可以用、以及具体怎么调用。

```txt
用法：scripts/process.py [可选参数] 输入文件

处理输入文件中的数据，并生成一个总结报告。

可选参数：
  --format FORMAT    指定输出格式：json、csv、table（默认是 json）
  --output FILE      把结果写到指定文件里，而不是直接显示在终端
  --verbose          把处理过程的信息输出到错误流 stderr

示例：
  scripts/process.py data.csv
  scripts/process.py --format csv --output report.csv data.csv
```

**尽量写得简短一些**，因为这些输出会和其他信息一起占用代理的上下文空间。



## 3. 写出清楚、对使用者有帮助的报错信息

**AI agent 报错后，会根据这条错误信息来决定下一步怎么改。**如果你只写一句模糊的 “Error: invalid input（输入无效）”，它其实很难知道该怎么调整，这样就等于浪费了一轮操作。更好的方式是直接告诉它：

* 具体哪里错了
* 正确要求是什么
* 接下来应该怎么改

```txt
错误：--format 必须是以下选项之一：json、csv、table。
当前收到的是："xml"
```



## 4. 让脚本输出有固定结构、方便读取的内容

相比随意写的一大段文字，更推荐让脚本**输出** JSON、CSV、TSV 这类**结构清晰的格式**。因为这种格式不但 AI 代理更容易读懂，像 `jq`、`cut`、`awk` 这些常见工具也能直接处理，所以脚本更容易和其他命令串起来一起用。

```txt
# 通过空格对齐——程序不容易解析
NAME          STATUS    CREATED
my-service    running   2025-01-15

# 使用分隔明确的格式——字段边界清晰，不会有歧义
{"name": "my-service", "status": "running", "created": "2025-01-15"}
```

**要把“真正的结果数据”和“提示、进度、警告、报错”分开输出。**
 **结构化结果放到 `stdout`，进度提示、警告和其他辅助信息放到 `stderr`。**
 **这样 AI 代理在读取结果时，就能拿到干净、方便解析的数据；如果需要排查问题，它也还能看到那些诊断信息。**



## 5. 后续注意事项

* **幂等性**：同一个命令执行一次和执行多次，结果最好一致。例如：“添加一个标签”如果重复执行，系统不应该报错或创建两个一样的标签，而是发现已有就跳过。
* **输入约束**：不要让系统去猜用户到底想要什么。例如用户说“给我创建一个状态”，这里“状态”可能是 `open`、`closed`、`pending`，这时候就应该报错提示“请输入允许的状态值”，而不是系统自己瞎猜。
* **支持 dry-run（演练模式）。** 对于具有破坏性或会修改状态的操作，提供一个 `--dry-run` 参数，可以让智agent 先**预览将会发生什么**，而不真正执行操作。
* **有意义的退出码。** 不同类型的失败应使用**不同的退出码**（例如：未找到、参数无效、认证失败），并且要在 `--help` 输出中写清楚这些退出码分别代表什么，这样 agent 才能知道每个退出码的含义。
* **安全默认值**。默认不要“太激进”。比如删除数据库、覆盖文件、批量修改内容，这种高风险操作，最好不能直接执行，而是要用户额外加上：`--confirm`、`--force`，这样可以防止误操作。
* **可预测的输出大小**。agent 一次能看到的输出通常不是无限的，太长会被截断。所以脚本不要默认吐出一大堆内容，如果输出内容很大且不适合分页，就要求智能体必须传入 `--output` 参数，用来指定输出到某个文件，或者传入 `-` 明确表示输出到标准输出（stdout）。
