如何通过一边测试、一边改进的方法，判断你的技能输出效果好不好。

你写好了一个技能，也拿一个提示词试过了，看起来好像能用。
 但它真的稳定吗？换几种不同的提问方式还行不行？遇到边界情况会不会出错？和“不用这个技能”相比，效果到底有没有更好？

这时候，**做一套有结构的评测（evals）** 就能帮你回答这些问题。
 它还能让你形成一个 **测试 → 发现问题 → 改进技能 → 再测试** 的反馈循环，这样你就能更系统地把这个技能越改越好。



# 1. 设计测试用例

一个**测试用例**通常有三个部分：

* **提示词（Prompt）**：一条真实用户可能会输入的消息，也就是别人实际会怎么问。
* **预期输出（Expected output）**：用人能看懂的话，说明“什么样的结果才算成功”。
* **输入文件（可选，Input files）**：这个技能运行时可能需要用到的文件。

你需要把这些测试用例，统一放到技能文件夹中的 `evals/evals.json` 这个文件里。

`evals/evals.json`:

```txt
{
  "skill_name": "csv-analyzer",
  "evals": [
    {
      "id": 1,
      "prompt": "我有一份每月销售数据的 CSV 文件，在 data/sales_2025.csv。你能帮我找出收入最高的 3 个月，并做一个柱状图吗？",
      "expected_output": "生成一张展示收入最高 3 个月的柱状图，坐标轴和数值标注清晰。",
      "files": ["evals/files/sales_2025.csv"]
    },
    {
      "id": 2,
      "prompt": "我的下载目录里有一个叫 customers.csv 的文件，有些行缺少邮箱地址——你能帮我清理一下，并告诉我一共有多少条缺失吗？",
      "expected_output": "输出一个清理后的 CSV 文件，妥善处理缺失邮箱，并说明缺失邮箱的数量。",
      "files": ["evals/files/customers.csv"]
    }
  ]
}
```

**如何写出好的测试提示词**：

* **先从 2 到 3 个测试用例开始**。在你还没看到第一轮测试结果之前，不要一下子投入太多。后面如果有需要，再慢慢把测试集扩充就行。

* **测试提示词要尽量多样化**。要使用不同的表达方式、详细程度和语气风格。比如：

  * 有些提示词可以更随意一点，像：“**嘿，你能帮我清理一下这个 csv 吗？**”
  * 有些提示词可以更准确、更具体一点，像："**解析 `data/input.csv` 这个 CSV 文件，删除 B 列为空的行，并把结果写入 `data/output.csv`。**"

* **不要只测“最正常、最理想”的情况**（考虑边界情况）。至少要加一个测试提示词，专门用来测试一些特殊场景，比如：

  * 输入格式有问题
  * 用户提出了比较少见、比较奇怪的要求
  * 技能说明本身可能不够明确，容易让模型理解有歧义的情况

  还要故意测一些“容易出问题”的情况，看看这个技能到底稳不稳。

* **要使用真实一点的上下文**。真正的用户在提问时，通常会提到**文件路径、列名，或者一些和自己场景有关的信息**。像 **“处理一下这些数据”** 这种说法就太模糊了，基本测不出什么有价值的东西。

先不用着急定义特别具体的通过/失败判断标准。现在先把 **提示词** 和 **预期输出** 写好就行。等你看到第一轮运行结果之后，再去补充更详细的检查规则。

> **第一步先把测试题和理想结果准备好，先跑起来。第二步再根据实际输出，补更细的判断标准。**



# 2. 执行评测

核心做法是：每个测试用例都跑两遍。

一遍**使用这个技能**来跑，另一遍**不使用这个技能**（或者用这个技能的旧版本）来跑。

这样做的目的，是为了给你一个**对照基线**，方便你比较效果到底有没有变好。



## 1. 工作区目录结构

把评测结果整理到一个 **workspace（工作区）目录** 里，这个目录和你的技能目录放在一起。每完整跑一轮评测，都单独建一个 **iteration-N/** 目录来存放结果。这里的 **N** 表示第几轮迭代，比如 `iteration-1/`、`iteration-2/`。

在每一轮迭代目录里面，**每个测试用例也都要有自己单独的评测目录**，并且这个目录下还要分成：

* **with_skill/**：使用技能时的结果
* **without_skill/**：不使用技能时的结果

```txt
csv-analyzer/
├── SKILL.md
└── evals/
    └── evals.json
csv-analyzer-workspace/
└── iteration-1/
    ├── eval-top-months-chart/
    │   ├── with_skill/
    │   │   ├── outputs/       # 本次运行生成的文件
    │   │   ├── timing.json    # 消耗的 Token 和运行时长
    │   │   └── grading.json   # 检查结果
    │   └── without_skill/
    │       ├── outputs/
    │       ├── timing.json
    │       └── grading.json
    ├── eval-clean-missing-emails/
    │   ├── with_skill/
    │   │   ├── outputs/
    │   │   ├── timing.json
    │   │   └── grading.json
    │   └── without_skill/
    │       ├── outputs/
    │       ├── timing.json
    │       └── grading.json
    └── benchmark.json         # Aggregated statistics
```

你**手动编写**的主要文件是 `evals/evals.json`。

其他那些 JSON 文件，比如：

* `grading.json`
* `timing.json`
* `benchmark.json`

都是在评测过程中生成的。它们可能是由 **agent 自动生成**、**脚本生成**，或者 **你自己补充生成** 的。



## 2. 启动运行任务

每一次评测运行，都应该在一个**干净的上下文环境**里开始。也就是说，**不要带着前一次运行残留的状态**，也不要带着你开发这个技能时留下的上下文信息。

这样才能保证：agent 真正依据的，只有 `SKILL.md` 里写的内容。

如果你用的是支持 **subagents（子代理）** 的环境（比如 Claude Code），这种隔离通常是天然就有的，因为每个子任务开始时，都会是一个新的、干净的环境。

如果你的环境**不支持子代理**，那就要为**每一次运行单独开一个新会话**。

对于每一次运行，你都要提供这些信息：

* **技能路径**（如果是做基线对比，那就不提供技能）。这次要不要用某个技能，用的话它在哪
* **测试提示词**。这次测试要问什么
* **所有输入文件**。需要用到哪些文件
* **输出目录**。最终结果要保存到哪里

下面是一个示例，展示了在“使用技能”的单次运行中，你可以如何给 agent 下达指令：

```txt
执行下面这个任务：
- 技能路径：/path/to/csv-analyzer
- 任务内容：我有一份每月销售数据的 CSV 文件，在 data/sales_2025.csv。
你能帮我找出收入最高的 3 个月，并做一个柱状图吗？
- 输入文件：evals/files/sales_2025.csv
- 输出保存到：csv-analyzer-workspace/iteration-1/eval-top-months-chart/with_skill/outputs/
```

做对照组的时候，**题目还是同一个题目**，只是这次**不用这个技能**，然后把输出结果放到 `without_skill/outputs/` 里面，方便后面和“用了技能”的结果做比较。

如果你本来就已经有一个技能，现在只是想把它变得更好，那你真正要比较的不是“有技能 vs 没技能”，

而是：**新版本技能 vs 旧版本技能**

所以在你开始修改之前，先把旧版本备份一份，比如：

```bash
cp -r <skill-path> <workspace>/skill-snapshot/
```

然后在做对照测试时，让这次运行指向这个**备份下来的旧版本技能**，并且把输出结果保存到 `old_skill/outputs/`，而不是 `without_skill/`。



## 3. 记录耗时数据

**耗时数据**能帮助你比较：这个技能和旧版本相比，到底多花了多少时间、多少 token。

因为不同技能的取舍是不一样的：

* 有些技能虽然**大幅提升了输出质量**，但 **token 消耗变成了原来的 3 倍**
* 有些技能则是**效果更好，同时成本还更低**

这两种情况的意义是完全不一样的。

所以，当每一次运行结束后，你都应该记录下：

* **本次消耗的 token 数量**
* **本次运行花费的时间**

`timing.json`：

```txt
{
  "total_tokens": 84852,
  "duration_ms": 23332
}
```

在 **Claude Code** 里，当一个 **subagent（子代理）任务**完成时，任务完成通知（[task completion notification](https://platform.claude.com/docs/en/agent-sdk/typescript#sdk-task-notification-message)）中会带上两个信息：

* **total_tokens**：总共消耗了多少 token
* **duration_ms**：任务总耗时多少毫秒

你要在这个时候**立刻把这两个值保存下来**，因为它们**不会被保存到别的地方**。



# 3. 写检查规则

**断言**就是一些**可以被检查、可以被验证的规则**，用来说明输出结果**应该包含什么**，或者**应该做到什么程度**。这些断言最好在你**看到第一轮输出结果之后再补充**，因为很多时候，在技能真正跑过之前，你其实并不完全知道“什么样才算好”。

**好的断言**：

* **“输出文件是有效的 JSON”** —— 可以通过程序直接验证
* **“柱状图的坐标轴有标签”** —— 条件具体，而且能直接观察出来
* **“报告里至少包含 3 条建议”** —— 可以明确统计数量

不好的断言通常是这样：

* **“输出结果很好”** —— **太模糊了**，根本没法判断怎么才算通过
* **“输出里必须一字不差地出现 `Total Revenue: $X` 这句话”** —— **太死板了**；就算结果是对的，只是换了一种说法，也会被误判为失败

**不是所有东西都适合写成断言**。有些质量维度，比如：

* **写作风格**
* **视觉设计**
* **输出整体看起来是不是“感觉对”**

这些东西很难拆成简单的“通过 / 不通过”规则。这种情况更适合让**人来人工检查**（[human review](https://agentskills.io/skill-creation/evaluating-skills#reviewing-results-with-a-human).）。所以，**断言应该留给那些可以客观检查的内容**。

> **能用明确标准检查的，就写成断言**。**很依赖主观感觉判断的，就靠人工 review**。

**你需要在 `evals/evals.json` 文件的每一条测试用例下面，继续补上对应的断言规则。**

`evals/evals.json`：

```txt
{
  "skill_name": "csv-analyzer",
  "evals": [
    {
      "id": 1,
      "prompt": "我有一份每月销售数据的 CSV 文件，在 data/sales_2025.csv。你能帮我找出收入最高的 3 个月，并做一个柱状图吗？",
      "expected_output": "生成一张展示收入最高 3 个月的柱状图，图中坐标轴和数值标注清晰。",
      "files": ["evals/files/sales_2025.csv"],
      "assertions": [
        "输出结果中包含一个柱状图图片文件",
        "图表中正好展示了 3 个月份",
        "横轴和纵轴都有标签",
        "图表标题或说明中提到了 revenue（收入）"
      ]
    }
  ]
}
```



# 4. 给输出结果打分

**评分（Grading）** 的意思是：把每一条**断言**都拿去和**实际输出结果**对照检查，然后记录它是 **PASS（通过）** 还是 **FAIL（失败）**，并且写清楚**具体证据**。

这里说的“证据”，应该是**引用输出中的实际内容**，或者明确指出输出里的对应位置， 而不只是简单写一句自己的看法。比如：

* PASS：输出目录中确实生成了 `top_3_months_chart.png` 文件
* PASS：图表标题写着 ‘Top 3 Months by Revenue’
* FAIL：图中只显示了 2 个月，不满足“正好 3 个月”的要求

**评分不是凭感觉，而是要拿实际结果当证据来判断**。

最简单的方法是：把**输出结果**和**断言规则**一起交给一个 LLM，让它逐条判断每条断言是否成立。不过，如果某些断言是那种**可以通过代码直接检查**的内容，比如：

* JSON 是否合法
* 行数是否正确
* 文件是否存在、尺寸是否符合预期

那就更适合用**验证脚本**来检查。因为对于这种机械性的判断，**脚本通常比 LLM 更可靠**，而且这些脚本还能在后面的多轮迭代里反复复用。

`grading.json`：

```txt
{
  "assertion_results": [
    {
      "text": "输出结果中包含一个柱状图图片文件",
      "passed": true,
      "evidence": "在 outputs 目录中找到了 chart.png 文件（45KB）"
    },
    {
      "text": "图表中正好展示了 3 个月份",
      "passed": true,
      "evidence": "图表中显示了 3 个柱子，分别对应 March、July 和 November"
    },
    {
      "text": "横轴和纵轴都有标签",
      "passed": false,
      "evidence": "Y 轴标注为 'Revenue ($)'，但 X 轴没有标签"
    },
    {
      "text": "图表标题或说明中提到了收入",
      "passed": true,
      "evidence": "图表标题是 'Top 3 Months by Revenue'"
    }
  ],
  "summary": {
    "passed": 3,
    "failed": 1,
    "total": 4,
    "pass_rate": 0.75
  }
}
```

## 1. 评分原则

* 要求判定为 **PASS** 时必须有**明确、具体的证据**。不要“从宽判断”或“默认它算通过”。如果某条断言写的是 **“includes a summary（包含总结）”**，而输出里只是有一个标题叫 **“Summary”**，下面只有一句很空泛的话，那也应该判定为 **FAIL** —— 因为它只是**形式上有这个标签**，但**内容实质并没有满足要求**。

* 也要**审查这些断言本身**，不要只看最终结果。在评分时，要留意这些情况：

  * **断言太容易**：不管技能质量好不好，几乎都会通过。
  * **断言太苛刻**：就算输出已经很好了，也还是很容易失败。
  * **断言无法验证**：光看输出内容，根本没办法判断它是否满足。

  如果发现这类问题，就要在**下一轮迭代**里把这些断言修改好。

> 对于比较两个技能版本，可以尝试使用**盲评（blind comparison）**：
>
> 把两个输出结果都交给一个 LLM 评审，但**不要告诉它哪个结果来自哪个版本**。评审模型按照它自己的评分标准，从整体上评价这些质量，比如：
>
> * 组织结构
> * 格式呈现
> * 实用性
> * 精致度
>
> 这样做可以避免它因为先入为主地觉得“某个版本应该更好”而产生偏见。
>
> 这种方法可以作为**断言评分（assertion grading）的补充：
>  因为有时候，两个输出虽然都通过了所有断言，但它们在整体质量**上仍然可能有很大差别。



# 5. 对结果进行汇总

当这一轮迭代中的**每一次运行**都评分完成后，不要只盯着单个结果看了，而是要把这一轮的成绩**统一做个总表**。
要**按不同配置分别计算汇总统计数据**，并把这些数据保存到与各个评估目录同级的 `benchmark.json` 文件中
 （例如：`csv-analyzer-workspace/iteration-1/benchmark.json`）。

`benchmark.json`:

```txt
{
  "运行结果汇总": {
    "使用技能": {
      "通过率": { "平均值": 0.83, "标准差": 0.06 },
      "耗时（秒）": { "平均值": 45.0, "标准差": 12.0 },
      "Token 数": { "平均值": 3800, "标准差": 400 }
    },
    "未使用技能": {
      "通过率": { "平均值": 0.33, "标准差": 0.10 },
      "耗时（秒）": { "平均值": 32.0, "标准差": 8.0 },
      "Token 数": { "平均值": 2100, "标准差": 300 }
    },
    "差值": {
      "通过率": 0.50,
      "耗时（秒）": 13.0,
      "Token 数": 1700
    }
  }
}
```

`delta`（差值）告诉你两件事：

1. 这个技能**付出了什么代价**（比如更花时间、更多 token），
2. 它**换来了什么收益**（比如更高的通过率）。

如果一个技能虽然多花了 **13 秒**，但能让通过率提高 **50 个百分点**，那它大概率是值得的。

但如果一个技能让 token 用量**翻倍**，却只带来 **2 个百分点** 的提升，那它可能就不值得。

> 标准差（`stddev`）只有在**每个评估做了多次运行**时才真正有意义。在前期迭代里，如果你只有 **2 到 3 个测试用例**，而且每个用例只跑了 **1 次**，那就先重点看：
>
> * **原始通过数量**
> * **delta（差值）**
>
> 因为这时候那些统计指标还不太有参考价值。等到你后面把**测试集扩大**了，并且让**每个评估都运行多次**之后，这些统计指标才会开始变得有用。



# 6. 分析规律

汇总统计数据有时候会**掩盖一些重要规律**。所以在算完 benchmark 之后，还要继续检查：

* 把那些在**两种配置下都总是通过**的断言**删掉**，或者换成更有区分度的断言。因为这类断言其实**提供不了什么有价值的信息**——说明就算**不用这个技能**，模型本身也能轻松做到。这样的断言会让“**使用技能**”那一边的通过率看起来更高，但这种提升**并不真正体现技能的价值**。

* 检查那些在**两种配置下都总是失败**的断言。这种情况通常说明几种可能：这个断言本身有问题（要求模型做到它根本做不到的事）、这个测试用例太难了、这个断言检查的方向不对，测错了东西。在进入下一轮迭代之前，要先把这些问题修好。

* 重点研究那些 **用了技能就通过，不用技能就失败** 的断言。这类地方最能说明：**这个技能确实带来了明显价值。**你要进一步弄清楚原因——到底是**哪一条指令**，或者**哪个脚本**，让结果产生了这种差别。

* 如果同一个测试，这次能过，下次又不过，来回不稳定，那你就要把说明写得**更清楚、更具体**。因为这种情况通常说明两件事里的其中一种：

  * 第一，这个测试本身就不稳定，模型每次回答时会有点随机，所以结果忽上忽下。
  * 第二，你给模型的说明太模糊了，它这次这样理解，下次又那样理解，所以做出来的结果不一样。

  那该怎么改呢？就是**把要求写得更明确一点**，比如：

  * 少说空话，多写清楚到底要做什么
  * 给出具体例子
  * 告诉它什么算对，什么不算对
  * 把容易让人误解的地方补清楚

* 检查那些**耗时特别长**、或者 **token 用量特别高**的异常情况。如果某个测试用例花的时间是其他评估的 **3 倍**，就去查看它的**执行记录**（也就是模型在这次运行过程中完整做了什么的日志），找出卡在哪里、慢在哪里。

# 7. 让人来复查结果

断言评分和规律分析确实能发现很多问题，但它们只能检查**你事先想到并写进断言里的那些点**。

人工复查能带来一个新的视角——它可以发现那些你一开始没预料到的问题，也能看出有些输出虽然**表面上 technically correct（技术上算对）**，但其实**没有抓住重点**，或者发现一些很难用“通过 / 不通过”这种方式表达的问题。所以，对每一个测试用例，都要把**实际输出内容**和**评分结果**放在一起看，做一次人工检查。

为每个测试用例记录**具体的反馈意见**，并把这些反馈保存在工作目录里，（例如，和各个评估目录放在一起，保存成一个 `feedback.json` 文件）。

`feedback.json`:

```txt
{
  "eval-top-months-chart": "图表缺少坐标轴标签，而且月份是按字母顺序排列的，不是按时间顺序排列的。",
  "eval-clean-missing-emails": ""
}
```

**“图表缺少坐标轴标签”** 这种反馈是有用的，因为别人一看就知道该改什么；但 **“看起来不太好”** 这种说法太笼统，别人不知道到底问题出在哪，也就没法下手修改。

如果反馈是空的，就说明这份输出整体看起来没什么明显问题，也就是说，这个测试用例通过了你的人工检查。

所以在这一轮优化里，你应该把主要精力放在那些**你已经提出了明确问题**的测试用例上，优先修改这些真正暴露出具体问题的地方。

# 8. 继续迭代优化这个技能

在完成评分和人工检查之后，你手里通常会有三类很有用的信息：

* **没通过的检查项**：告诉你“错在哪”
* **人工反馈**：告诉你“整体上哪里不行”
* **执行记录**：告诉你“它是怎么一步步错成这样的”

把这些信息真正转化成对技能的改进，**最有效的方法**通常是：

把前面这三类信息，**连同当前的 `SKILL.md` 一起**交给一个 LLM，然后让它提出应该怎么修改。

因为 LLM 可以把这些线索综合起来看——把**没通过的检查项**、**人工反馈里的抱怨**、以及**执行记录里暴露出的行为问题**放在一起分析，找出其中的共同规律。

这些关联如果靠人手动去一条条对照，通常会很费劲、也很麻烦。所以，当你去提示这个 LLM 的时候，要把下面这些指导原则也一起带上：

* 要根据这些反馈，去发现和解决**更根本、更通用的问题**，而不是只对眼前这几个测试例子头痛医头、脚痛医脚。为这个技能以后面对的会是各种不同的提问，不会永远只是在做这几道测试题。所以你在修改时，重点应该放在**把底层问题改对、改稳**，让它在更多场景下都更好用，而不是只为了让某几个例子通过，临时加一些很局部、很死板的修补。
* **让这个技能尽量保持简洁，不要越写越臃肿**。很多时候，几条真正关键、写得清楚的说明，比一长串事无巨细的规则更管用。如果执行记录里显示模型做了很多**没必要的事**，比如多余的检查、没必要的中间结果输出，那就把相关说明删掉。如果你发现规则越加越多，通过情况却不再提升，那说明这个技能可能已经被限制得太死了。这时候不妨反过来试试：**删掉一些说明**，看看结果能不能保持不变，甚至变得更好。
* 把“**为什么要这么做**”也写出来。相比那种很死板的命令式说法——比如“**永远都要做 X，绝对不要做 Y**”——带有原因的说明通常更有效，比如“**做 X，因为 Y 往往会导致 Z 问题**”。因为模型在**明白目的**的情况下，往往会更稳定、更准确地照着要求去做。
* 把那些**反复重复做的工作打包进去**。如果你发现每次测试运行时，都会单独写一个差不多的辅助脚本，比如画图脚本、数据解析脚本，那这就是一个信号：说明这类东西应该直接放进这个技能的 `scripts/` 目录里，作为技能的一部分复用。关于具体怎么做，可以去看 **[Using scripts](https://agentskills.io/skill-creation/using-scripts)** 那一节。

## 1. 整个迭代流程

1. 把评估过程中得到的这些信号，连同当前的 `SKILL.md`，一起交给一个 LLM，让它提出改进建议。
2. 检查这些修改建议，并把合适的改动应用进去。
3. 在一个新的 `iteration-<N+1>/` 目录里，把所有测试用例重新跑一遍。
4. 对新的结果进行评分，并把它们汇总起来。
5. 让人再检查一遍。然后继续重复这个流程。

当你对结果已经满意、反馈基本上一直都是空的，或者连续几轮迭代之间已经看不到明显改进时，就可以停下来了。

> [`skill-creator`](https://github.com/anthropics/skills/tree/main/skills/skill-creator)这个 Skill 可以把这整套流程里的很多工作自动化掉，比如：运行评估、给断言打分、汇总 benchmark 结果、把结果整理出来，方便人来复查
