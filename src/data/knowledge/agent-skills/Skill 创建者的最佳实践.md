如何编写**范围清晰、并且与任务匹配得当**的 skills。



# 1. 从真正有价值的专业经验出发

创建 skill 时，一个常见误区是：让 LLM 直接生成 skill，却没有提供任何**领域特定的上下文**，而只是依赖它在训练中学到的一般性知识。这样产生的结果往往会很空泛、很通用，比如“妥善处理错误”“遵循认证最佳实践”之类，而不是那些真正让 skill 有价值的具体内容，例如**特定 API 的使用模式、边界情况处理，以及项目内部约定**。

高质量的 skill 建立在**真实专业经验**之上。关键在于：在创建过程中，把**领域特定的上下文**提供进去。

## 1. 从真实的实操任务中提炼 skill

先和 agent 通过对话完成一次**真实任务**，在过程中持续提供上下文、纠正信息和你的偏好。然后再把其中**可复用的模式**提炼成一个 skill。请重点关注：

* **有效的步骤** —— 哪一套操作顺序最终成功解决了问题
* **你做出的纠正** —— 你在哪些地方调整了 agent 的做法（例如：“用库 X，不要用 Y”“记得检查边界情况 Z”）
* **输入 / 输出格式** —— 数据传入时是什么样，输出后又是什么样
* **你提供的上下文** —— agent 原本并不知道的、与项目相关的事实、约定或限制条件



## 2. 从现有的项目产物中归纳提炼

当你已经积累了**一批现成知识**时，可以把这些内容提供给 LLM，让它帮你**归纳生成一个 skill**。比如，一个**基于你们团队真实事故报告和运行手册提炼出来的数据流水线 skill**，通常会比一个仅仅根据通用“数据工程最佳实践”文章总结出来的 skill 更好用，因为前者能够真正覆盖你们自己的 **schema、故障模式以及恢复流程**。关键在于：要依赖**项目特定的材料**，而不是泛泛的通用参考资料。

好的来源材料包括：

* 内部文档、运行手册和风格指南
* API 规范、schema 和配置文件
* 代码评审意见与问题跟踪记录（可以反映反复出现的关注点，以及评审者的预期）
* 版本控制历史，尤其是补丁和修复记录（能通过真实改动看出其中的规律）
* 真实世界中的故障案例，以及对应的解决方法



# 2. 在真实运行中持续打磨优化

skill 的第一版通常都还需要继续打磨。把它拿去处理真实任务，然后把结果——**包括所有结果，不只是失败案例**——再反馈回创建过程中。你可以追问这些问题：

- 是什么导致了**误触发**？
- 有哪些内容**漏掉了**？
- 有哪些东西其实**可以删减**？

哪怕只做一轮“先执行、再修订”，质量通常都会有明显提升；而在复杂领域里，往往做上好几轮效果会更好。

> 不要只看最终输出，还要看 agent 的**执行轨迹（execution traces）**。如果 agent 在一些没有产出的步骤上浪费了时间，常见原因通常有这些：
>
> * **指令太模糊**：agent 会不断尝试好几种方法，才找到真正可行的一种.
>
> * **指令不适用于当前任务**：即使当前场景并不合适，agent 还是照着做了.
>
> * **给出的选项太多，却没有明确默认方案**：agent 不知道优先选哪一个，容易来回试探

如果你想用一种**更结构化的方式**来迭代优化——包括测试用例、断言和评分机制——
可以参考 **[Evaluating skill output quality](https://agentskills.io/skill-creation/evaluating-skills)**。



# 3. 合理使用上下文

一旦某个 skill 被激活，它完整的 `SKILL.md` 正文就会和**对话历史、系统上下文以及其他已激活的 skills** 一起被加载进 agent 的上下文窗口中。你在 skill 里写下的每一个 token，都会和这个窗口里的其他所有内容一起“竞争” agent 的注意力。



## 1. 补充 agent 不知道的内容，省去它已经知道的内容

重点写那些 **agent 如果没有你的 skill 就不会知道的内容**：比如项目特有的约定、领域特定的流程、不明显的边界情况，以及应该使用的具体工具或 API。

你不需要去解释什么是 PDF、HTTP 是怎么工作的，或者数据库迁移是做什么的。

```txt
<!-- 过于冗长——agent 本来就知道什么是 PDF -->
## 提取 PDF 文本

PDF（便携式文档格式）是一种常见的文件格式，里面可以包含
文本、图片以及其他内容。要从 PDF 中提取文本，你需要
使用一个库。推荐使用 pdfplumber，因为它在大多数情况下都处理得不错。

<!-- 更好的写法——直接切入 agent 本来不会凭空知道的信息 -->
## 提取 PDF 文本

使用 pdfplumber 提取文本。对于扫描版文档，降级改用
pdf2image 搭配 pytesseract。

```python
import pdfplumber

with pdfplumber.open("file.pdf") as pdf:
    text = pdf.pages[0].extract_text()
```

对 skill 里的每一部分内容，都问自己一句：**“如果没有这条指令，agent 会不会在这里做错？”**

如果删掉这条指令，对结果没有影响，那就果断删除它！

**如果你拿不准，就去测试。**

而且，如果 agent 在**没有这个 skill 的情况下**也已经能把整个任务完成得很好，那这个 skill 可能并没有真正增加价值。至于如何系统化地做这类测试，可以参考 **[Evaluating skill output quality](https://agentskills.io/skill-creation/evaluating-skills)**。



## 2. 把 skill 设计成连贯的功能单元

决定一个 skill 应该覆盖哪些内容，就像决定一个函数该做什么一样：你希望它能封装一个**连贯且完整的工作单元**，并且能够和其他 skill 很好地组合使用。

如果一个 skill 的范围**太窄**，那么完成一个任务时可能就得同时加载多个 skill，这样不仅会增加开销，还可能带来相互冲突的指令。 如果一个 skill 的范围**太宽**，又会导致它很难被准确触发。

比如：

- 一个用于**查询数据库并格式化结果**的 skill，可以算作一个连贯的单元
- 但如果同一个 skill 还要同时负责**数据库管理**，那它多半就管得太多了



再比如：

1. **skill 的范围太小，不好。**

   如果你把一个任务拆得过细，就会出现：完成一件事要同时激活很多个 skill。

   比如**本来“查询数据库并整理结果”是一整套连续动作**，结果你拆成：

   * 一个 skill 只负责连数据库
   * 一个 skill 只负责写 SQL
   * 一个 skill 只负责格式化结果

   这样 agent 做一次数据库查询，**可能要同时加载 3 个 skill**。问题是：

   * 更麻烦
   * 更耗上下文
   * **不同 skill 之间还可能互相打架**

2.  **skill 的范围太大，也不好。**

   如果一个 skill 什么都想管，它就会变得不够精准。

   比如一个 skill 既负责：

   * 查询数据库
   * 格式化结果
   * 数据库备份
   * 用户权限管理
   * 数据库性能调优

   那这个 skill 就太宽了。这样 agent 很难判断：**什么时候才该激活它？**

   因为用户只是想“查个数据”，结果你这个 skill 却把“数据库管理员”的工作也全塞进去了。

> **一个 skill 最好只解决一类完整问题，而不是只做半步，也不是把很多不相干的事全包进去。**



## 3. 细节不要过多也不要过少

**过于面面俱到的技能，带来的伤害可能比帮助还大**—— agent 很难提炼出真正相关的内容，还可能被那些其实不适用于当前任务的指令带偏，**走上一些低效甚至无用的路径**。

相比之下，**简洁、分步骤的指导，再配上一个可运行的示例**，通常比事无巨细的长篇文档效果更好。

当你发现自己开始试图覆盖**所有边界情况时**，**不妨停下来想一想**：**这里面大多数情况，是不是其实更适合交给代理自己判断和处理。**



## 4. 用“渐进式展开”的方式来组织大型技能。

规范建议将 `SKILL.md` 控制在 **500 行以内、5000 token 以内**——只保留代理在每次运行时都必须看到的核心指令。如果某个技能确实需要更多内容，就应当把那些更详细的参考资料拆分到单独的文件里，比如放进 `references/` 或类似的目录中。

关键在于：你要明确告诉代理**什么时候**去加载哪个文件。

**比如，“如果 API 返回的不是 200 状态码，就去阅读 `references/api-errors.md`”**

就比笼统地说一句 “更多细节请看 `references/`”，要有用得多。

这样一来，代理就可以在**真正需要的时候**再按需加载上下文，而不是一开始就把所有内容都读进来——这正是“渐进式展开”的核心做法。



# 4.  把控“控制”的分寸

**不是每一步都要把 AI 管得死死的。**

有些事情本来就很简单、很稳，给它一个大方向就够了；

但有些事情很容易做错，或者一旦错了后果很明显，那就要把要求写得更具体、更死一点。

你可以把它理解成：

* **简单、不容易出错的事** → 少规定一点，让它自己发挥
* **复杂、容易翻车的事** → 多规定一点，告诉它必须怎么做

比如：

你让 AI “写一个活动宣传文案”，这个任务不算太脆弱，风格上可以灵活一点，所以只要告诉它语气、目标用户、长度就行，不用规定每一句怎么写。

但如果你让 AI “修改数据库里的生产环境配置”，**这个事情很容易出问题**，那就不能只说个大概，而要**明确写清楚**：先备份、再检查环境、只能改哪几个字段、改完怎么验证。

## 1. 任务越容易出错，要求就要越具体。

当一个任务有多种可行做法，而且允许结果存在一定差异时，就应该给 agent 更多自由度。

对于这类比较灵活的指令，**解释“为什么要这样做”**，往往比生硬规定“必须怎么做”更有效——因为代理一旦理解了指令背后的目的，就更能根据具体上下文做出更合适的判断。

比如，一个**代码审查技能**，可以告诉代理“应该重点关注什么问题”，而不一定要硬性规定“必须按哪几个固定步骤来审查”：

```txt
## 代码审查流程
1. 检查所有数据库查询是否存在 SQL 注入 风险（要使用参数化查询）
2. 验证每个接口是否都做了 身份认证检查
3. 查找并发代码路径中是否存在 竞态条件
4. 确认错误信息不会泄露 内部细节
```

当操作本身比较脆弱、结果的一致性很重要，或者某个步骤顺序必须严格遵守时，就应该写得更具规定性、更明确一些：

````txt
## 数据库迁移
严格按下面这个顺序执行：

```bash
python scripts/migrate.py --verify --backup
```

不要修改这条命令，也不要额外添加任何参数。
````

一个技能里，通**常有些地方要管严一点，有些地方可以放松一点**。**不要一刀切**，而是要根据每个部分的特点，分别决定控制力度。

## 2. 提供默认方案，而不是丢给它一堆选项。

当有多种工具或做法都能解决问题时，应该**先选一个默认方案**，并且只**顺带简要提一下其他可选方案**，而不是把它们当成完全同等的选项一起摆出来。

````txt
<!-- 选项太多 -->
你可以使用 pypdf、pdfplumber、PyMuPDF 或 pdf2image……

<!-- 清晰的默认方案，同时保留例外处理 -->
使用 `pdfplumber` 来提取文本：

```python
import pdfplumber
```

如果是需要 OCR 的扫描版 PDF，则改用 pdf2image 搭配 pytesseract。
````



## 3. 多告诉它具体怎么做，少只说空泛要求。

一个 Skill 应该教 agent **如何处理一类问题**，而不是只告诉它在某一个具体例子里该产出什么。

对比来看：

```txt
<!-- 只对这个特定任务有用的具体答案 -->
将 `orders` 表和 `customers` 表通过 `customer_id` 进行关联，
筛选 `region = 'EMEA'` 的数据，
然后对 `amount` 列求和。

<!-- 可复用的方法 —— 适用于各种分析类查询 -->
1. 先读取 `references/schema.yaml`，找到相关的数据表
2. 按照 `_id` 外键约定来关联各张表
3. 把用户请求里的筛选条件转换成 `WHERE` 子句
4. 根据需要对数值列做聚合，并将结果整理成 Markdown 表格
```

这并不意味着技能里**不能包含具体细节**——像**输出格式模板**（参见 [Templates for output format](https://agentskills.io/skill-creation/best-practices#templates-for-output-format)）、**“绝不能输出个人敏感信息（PII）”** 这样的约束，以及**针对某个工具的专门说明**，这些都很有价值。

重点在于：**即使其中包含一些具体细节，整体的方法也应该是可泛化、可复用的**，而不是只能应付某一个特定任务。



# 5. 有效指令的写法模式

这些是一些**可重复使用的技巧**，用来组织技能内容。

并不是每个技能都需要把这些方法全部用上——**只选择适合你当前任务的那些就可以了。**

## 1. 常见陷阱说明

很多技能里，**价值最高的内容**往往是一份“容易踩坑点”清单——也就是那些**和具体环境强相关、不能靠常理轻易猜到的事实**。

很多技能文档里，**最有用的部分，其实不是大道理，而是“这里特别容易翻车”的提醒。**

因为 AI 很多时候会按照“正常情况”“常规理解”去做事。

但真实项目里，经常会有一些**不按常理出牌的特殊情况**。

这些特殊情况，AI 靠自己猜，往往猜不到，或者很容易猜错。

所以你最应该写进技能里的，常常不是那种空话，比如：

- 要注意错误处理
- 要保证代码质量
- 要考虑边界情况

这些话虽然没错，**但太泛了**，AI 本来也知道一点，真正帮不到它太多。

更有价值的是这种内容：

- 这个项目里**不能用 `npm`，只能用 `pnpm`**
- 这个接口虽然是 `POST`，但其实**参数要放在 query 里**
- 这个目录名字看起来像源码目录，**其实是构建产物，不能改**
- 这个服务返回 `200` 也不一定成功，**要看 response body 里的 `success` 字段**
- 这个系统里用户 ID 表面叫 `id`，**但真正关联字段其实是 `user_id`**

这些就是“gotchas”——**不提前说，AI 很容易踩坑的隐藏规则**。

```txt
## 注意事项

- users 表使用的是软删除机制。查询时必须加上 WHERE deleted_at IS NULL，否则结果里会包含已经被停用的账号。
- 同一个用户 ID，在不同系统里的名字不一样：在数据库里叫 user_id，在认证服务里叫 uid，在计费 API 里叫 accountId。这三个其实指的是同一个值。
- /health 接口只要 Web 服务器还在运行，就会返回 200，即使数据库连接已经断了也一样。如果要检查服务是否真的整体正常，应该使用 /ready。
```

把这些“容易踩坑的点”直接放在 `SKILL.md` 里更好，因为 agent 会在真正遇到问题之前就先读到它们。

当然，你也可以把它们放到单独的参考文件里，但前提是你要明确告诉代理**什么时候去加载那个文件**。

只是对于那些**不明显、很反常识的问题**，代理往往未必能自己意识到“哦，现在该去读这个参考文件了”，所以就可能错过这些关键信息。

> 当 agent 犯了一个你**不得不出手纠正的错误时**，就把这个纠正内容补充到 **gotchas（易踩坑点）** 部分里。
>
> 这是迭代改进一个技能最直接的方法之一（参见 **通过真实执行来优化** [Refine with real execution](https://agentskills.io/skill-creation/best-practices#refine-with-real-execution)）。
>
> **通俗解释**：AI 哪里犯过错，你就把这个坑记下来，写进技能文档里。这样下次它再做类似任务时，就更不容易在同一个地方摔倒。



## 2.用于规定输出格式的模板

当你需要代理按**特定格式**输出内容时，最好直接提供一个**模板**。

这通常比用一大段文字去描述格式要求更可靠，因为代理更擅长照着**具体结构**去匹配和生成内容。

较短的模板可以直接写在 `SKILL.md` 里；

如果模板比较长，或者只在某些特定情况下才会用到，就可以把它放到 `assets/` 这类目录里，并在 `SKILL.md` 中说明何时引用它。这样只有在真正需要时才会加载。

````txt
## 报告结构

请使用下面这个模板，并根据具体分析内容对各部分做相应调整：

```markdown
# [分析标题]

## 执行摘要
[用一段话概述核心发现]

## 关键发现
- 发现 1，并附上支持数据
- 发现 2，并附上支持数据


## 建议
1. 具体、可执行的建议
2. 具体、可执行的建议
```
````

## 3. 多步骤工作流的检查清单

如果一件事要分很多步来做，最好把步骤列成清单。这样 AI 不容易做着做着就漏步骤，或者顺序搞错。

**清单的作用，不只是看起来整齐，而是能帮 AI 按顺序完成任务、随时知道自己做到哪了，并减少漏做和乱做。**

```txt
## 表单处理工作流

步骤：
1. 分析表单（运行 scripts/analyze_form.py）
2. 创建字段映射（编辑 fields.json）
3. 验证映射（运行 scripts/validate_fields.py）
4. 填写表单（运行 scripts/fill_form.py）
5. 校验输出结果（运行 scripts/verify_output.py）
```



## 4. 反复验证机制

要指示 agent 在进入下一步之前，先验证自己刚完成的工作。

这个模式就是：

**先完成工作 → 再运行验证手段（可以是脚本、参考检查清单，或者自检）→ 如果发现问题就修正 → 然后继续重复，直到验证通过为止。**

```txt
## 编辑工作流

1. 先进行你的修改
2. 运行验证：python scripts/validate.py output/
3. 如果验证失败：
	- 查看报错信息
	- 修复问题
	- 再次运行验证
4. 只有在验证通过后，才能继续下一步
```

参考文档也可以充当“验证器”——你可以指示 agent 在最终完成之前，先拿自己的结果去对照参考文档检查一遍。



## 5. 先规划，再验证，后执行。

遇到那种一次**会改很多东西**、或者**一旦做错就很难挽回的操作**，不要让 AI 上来就直接动手。要先让它写一份“准备怎么做”的计划，检查这份计划对不对，确认没问题后再真正执行。

```txt
## PDF 表单填写流程

1. 提取表单字段：运行 python scripts/analyze_form.py input.pdf → 生成 form_fields.json
（其中会列出每个字段的名称、类型，以及它是否为必填项）
2. 创建 field_values.json，把每个字段名映射到你准备填写的值
3. 进行验证：运行 python scripts/validate_fields.py form_fields.json field_values.json
（它会检查：填写的字段名是否都真实存在于表单中、类型是否匹配、必填字段是否有缺失）
4. 如果验证失败，就修改 field_values.json，然后重新验证
5. 填写表单：运行 python scripts/fill_form.py input.pdf field_values.json output.pdf
```

关键就在第 3 步：**用一个验证脚本，把计划（`field_values.json`）和真实依据（`form_fields.json`）对照检查。**

这样做的好处是，代理在真正填写 PDF 之前，就能先发现计划里有没有写错。

比如出现这样的报错：`Field 'signature_date' not found — available fields: customer_name, order_total, signature_date_signed`

这类错误信息就非常有用，因为它不仅告诉代理“你填的字段名不对”，还顺带告诉它“表单里实际可用的字段有哪些”。有了这些信息，代理就更容易自己修正错误，而不用每次都依赖人工介入。

## 6. 把可重复使用的脚本整理打包在一起。

当你在迭代一个技能时，可以对比 agent 在不同测试用例里的执行过程。

如果你发现 agent 每次运行时都在**重复“自己重新发明”同一套逻辑**——比如反复生成图表、解析某种固定格式、验证输出结果——这就是一个信号：
 你应该把这部分逻辑**正式写成一个经过测试的脚本**，然后放到 `scripts/` 目录里统一复用。

> 如果你发现 AI 每次做同类任务时，都在重复写一遍差不多的“临时小程序”或处理逻辑，那就别让它每次都现想现做了，干脆把这部分固定下来，写成正式脚本。

 关于如何设计并打包这些脚本的更多内容，请参见 **在技能中使用脚本**。 [Using scripts in skills](https://agentskills.io/skill-creation/using-scripts).
