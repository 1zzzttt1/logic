Agent Skills 的完整格式规范。



# 1. 目录结构

一个 skill 是一个目录，其中至少包含一个 `SKILL.md` 文件：（**必须强制叫 `SKILL.md`**，不能是其他，比如不能是`skill.md`、不能是`my-skill.md`等等）

```txt
skill-name/
├── SKILL.md          # 必需内容：元数据（用来描述这个 skill 本身的信息） + 说明内容
├── scripts/          # 可选项：可执行代码
├── references/       # 可选项: 说明文档
├── assets/           # 可选项：模板、资源
└── ...               # 任何额外的文件或目录
```



# 2. `SKILL.md`格式

`SKILL.md` 文件必须包含 **YAML frontmatter**，后面再跟着 **Markdown 内容**。



## 1.  Frontmatter

| 字段            | 是否必填 | 约束                                                         |
| --------------- | -------- | ------------------------------------------------------------ |
| `name`          | 是       | 最多 64 个字符。仅允许小写字母、数字和连字符。不能以连字符开头或结尾。 |
| `description`   | 是       | 最多 1024 个字符。不能为空。用于描述这个 skill 做什么，以及何时使用它。 |
| `license`       | 否       | 许可证名称，或对随附许可证文件的引用。                       |
| `compatibility` | 否       | 最多 500 个字符。用于说明环境要求（目标产品、系统软件包、网络访问等）。 |
| `metadata`      | 否       | 用于附加元数据的任意键值映射。                               |
| `allowed-tools` | 否       | 该 skill 可使用的预批准工具列表，以空格分隔。（实验性）      |

**最小示例：**

`SKILL.md`：

```markdown
---
name: skill-name
description: 描述这个技能能做什么，以及应在什么情况下使用它。
---
```



带有可选字段的示例：

```markdown
---
name: pdf-processing
description: 提取 PDF 文本、填写表单、合并文件。用于处理 PDF 时。
license: Apache-2.0
metadata:
  author: example-org
  version: "1.0"
---
```



### 1. `name` 字段

必填的 `name` 字段要求如下：

* 必须为 **1–64 个字符**
* 只能包含 **Unicode 小写字母数字字符**（`a-z`）和连字符 **`-`**
* **不能** 以 **`-`** 字符开头或结尾
* **不能** 包含连续的连字符 **`--`**
* **必须与父目录名称一致**。（比如父级目录叫 pdf-processing，这个 SKILL.md 文件里的 name 字段，必须也是 pdf-processing）



**有效示例**：

```txt
name: pdf-processing
```

```txt
name: data-analysis
```

```txt
name: code-review
```



**无效示例**：

```txt
name: PDF-Processing  # 不允许使用大写字母
```

```txt
name: -pdf  # 不能以 - 字符开头
```

```txt
name: pdf--processing  # 不允许出现连续的 - 字符
```



### 2. `description` 字段

必填的 `description` 字段要求如下：

* 必须为 **1–1024 个字符**
* 应同时说明 **该技能能做什么** 以及 **应在什么情况下使用**
* 应包含有助于代理识别相关任务的 **具体关键词**



**好的示例：**

```txt
description: 从 PDF 文件中提取文本和表格、填写 PDF 表单，并合并多个 PDF 文件。适用于处理 PDF 文档，或当用户提到 PDF、表单或文档提取时使用。
```



**差的示例**：

```txt
description: 用于处理 PDF。
```



### 3. `license` 字段

可选的 `license` 字段：

- 用于指定该技能所适用的许可证
- 建议保持简短（可以填写许可证名称，或随技能一并提供的许可证文件名称）

示例：

```txt
license: Proprietary. LICENSE.txt has complete terms
```



* **为什么需要 license 这个字段？**

  因为 **skill 本质上也是一种可分发、可复用的“作品”**。只要它会被别人使用、复制、修改、放进项目里，就会涉及一个问题：**别人能不能合法地用它**？**能不能修改**？**能不能商用**？是否必须保留原作者信息？

  `license` 字段就是用来回答这个问题的。

  `license` = **这个 skill 的使用规则说明**



### 4. `compatibility` 字段

可选的 `compatibility` 字段：

- 如果提供，长度必须为 **1–500 个字符**
- 只有当你的 skill 对运行环境有特定要求时，才建议包含该字段
- 可用于说明适用的目标产品、所需的系统软件包、网络访问需求等信息



示例：

```txt
compatibility: 适用于 Claude Code（或类似产品）
```

```txt
compatibility: Requires git, docker, jq, and access to the internet
```

```txt
compatibility: Requires Python 3.14+ and uv
```

> **大多数 skills 都不需要这个字段**



### 5. `metadata` 字段

可选的 `metadata` 字段：

- 是一个**从字符串键到字符串值的映射**
- 客户端可以用它来存储 **Agent Skills 规范中未定义的额外属性**
- 我们建议让键名具有一定的**唯一性**，以避免意外冲突



**示例**：

```txt
metadata:
  author: example-org
  version: "1.0"
```

**这个字段的作用**：

* name、description、license、compatibility。这些是 **规范里的正式字段**，可以直接给 agent 识别。
* **metadata**，是 **额外的自定义字段区域**，主要方便 **开发人员 / 平台方 / 客户端实现者** 存东西

举个例子：

假设有一个 skill：

```txt
---
name: vue-doc-helper
description: 帮助整理 Vue 组件文档和知识库文章结构。
metadata:
  myteam_author: "YM Z"
  myteam_version: "1.2.0"
  myteam_review_status: "approved"
---
```

* **对 agent 来说**：agent 主要看的是，name、description 两个字段。它会知道：这是个和 Vue 文档整理有关的 skill。

* **对开发者平台来说**，比如有一个平台是专门管理 skills 的（例如这个网站[The Agent Skills Directory](https://skills.sh/)），平台就会从 metadata 字段中读取一些关于 skills 的自定义信息，比如 skills 的作者、版本、审核状态等等。

  所以 `metadata` 的作用就出来了：**不是给用户对话时直接看的，而是给这种管理系统读取和使用的。**



### 6. `allowed-tools`  字段

可选的 `allowed-tools` 字段：

- 是一个**用空格分隔的工具列表**，这些工具被预先批准可运行
- 属于**实验性功能**
- 不同的 agent 实现对该字段的支持程度可能不同

**示例**：

```txt
allowed-tools: Bash(git:*) Bash(jq:*) Read
```



**例子：**

假设你写了一个 skill，专门用来处理 PDF 文件：

```txt
---
name: pdf-processing
description: 提取 PDF 文本、填写表单、合并 PDF 文件。适用于处理 PDF 文档时。
allowed-tools: read_file write_file run_shell_command
---
```

这里的：

```txt
allowed-tools: read_file write_file run_shell_command
```

意思就是：

这个 skill 在运行时，**预先允许使用这 3 个工具**：

* read_file
* write_file
* run_shell_command

并且它们之间是**用空格分隔**的，不是用逗号。

这个 skill 提前声明：为了完成任务，我通常需要用到这些工具，**请先把它们加入“已批准可用”的名单。**后续如果用到这些工具时，Agent 就不需要向你确认，它自己就会调用。（相当于给 Agent **提前授权 **使用某些工具）



## 2.  Body content

**frontmatter** 后面的 Markdown 正文**包含该 skill 的指令内容**。它**没有格式限制**。你可以编写任何有助于 agent 高效完成任务的内容。

推荐包含的部分：

- 分步骤说明
- 输入与输出示例
- 常见边界情况

请注意，一旦 Agent 决定激活某个 skill，它会加载整个文件。因此，如果 `SKILL.md` 内容较长，建议将部分内容拆分到被引用的其他文件中。

例子：

```markdown
---
name: markdown-fixer
description: 修复 Markdown 文档中的常见格式问题，如标题层级、代码块和列表格式。适用于整理教程、笔记或知识库文章时。
license: MIT
---

# Markdown 修复助手

当用户要求整理或修复 Markdown 文档时，使用这个 skill。

## 操作步骤

1. 先阅读全文。
2. 检查标题层级是否合理。
3. 检查代码块是否正确闭合。
4. 检查列表、链接和表格格式是否正常。
5. 在尽量不改动原意的前提下做最小修改。

## 注意事项

- 不要整篇重写内容
- 不要随意删除 frontmatter
- 不要修改代码含义

## 示例

用户说：  
“请帮我修复这篇 Markdown 教程的格式问题。”

你应该：
- 修复标题层级
- 修复代码块和列表格式
- 保留原文内容和语气
```



# 3. 可选的目录结构

## 1. scripts/

包含 agent 可运行的可执行代码。脚本应当：

- 保持自包含，或清晰说明其依赖项
- 提供有帮助的错误信息
- 妥善处理边界情况

支持的语言取决于具体的 agent 实现。常见选项包括 Python、Bash 和 JavaScript。



**举个例子：**

假设你做一个 skill：**清理 Markdown 文档**

除了 `SKILL.md` 说明文件，你还可以放一个 Python 脚本：

- 自动检查标题层级
- 找没闭合的代码块
- 修复一些常见格式问题

那这个脚本就应该做到：

- 不需要一堆隐藏依赖
- 出错时告诉你是哪一行有问题
- 如果文件为空，也别直接崩
- 明确说明“需要 Python 3”

**skill 里可以带可执行脚本，但脚本要写得清楚、稳妥、能落地。**



## 2 .references/

包含 agent 在需要时可读取的附加文档：

- `REFERENCE.md` —— 详细的技术参考文档
- `FORMS.md` —— 表单模板或结构化数据格式说明
- 领域专用文件（如 `finance.md`、`legal.md` 等）

应尽量让每个参考文件聚焦于单一主题。因为 agent 会按需加载这些文件，所以文件越小、越专注，占用的上下文就越少。

## 3. assets/

包含静态资源：

- 模板（文档模板、配置模板）
- 图片（示意图、示例图片）
- 数据文件（查找表、模式定义）



# 4. 渐进式披露

skill 应当按有利于高效使用上下文的方式来组织：

- **元数据（约 100 tokens）**：所有 skill 的 `name` 和 `description` 字段都会在启动时加载
- **说明内容（建议少于 5000 tokens）**：当 skill 被激活时，会加载整个 `SKILL.md` 正文
- **资源文件（按需加载）**：诸如 `scripts/`、`references/` 或 `assets/` 中的文件，只有在需要时才会被加载

建议将主 `SKILL.md` 控制在 **500 行以内**。更详细的参考资料应拆分到单独的文件中。



# 5. 对文件的引用

在你的 skill 里引用其他文件时，使用**相对于 skill 根目录的路径**：

`SKILL.md`：

```txt
See [the reference guide](references/REFERENCE.md) for details.

Run the extraction script:
scripts/extract.py
```

你在 `SKILL.md` 里引用别的文件时，尽量**直接引用**，不要绕很多层。

比如**推荐这样**：

```txt
请查看 `references/REFERENCE.md`
请运行 `scripts/extract.py`
```

不推荐这样：

```txt
先看 `references/INDEX.md`
然后在 INDEX.md 里再去找 `guides/setup.md`
再从 setup.md 里跳到 `details/config.md`
```

这就叫“深层引用链”，意思是：
 要找到真正有用的信息，得一层一层点进去，路径很绕，效率很低。

**在 `SKILL.md` 里尽量把重要文件直接指出来，别让代理或读的人顺着好多层文件慢慢找。**



# 6. 校验

使用 [skills-ref](https://github.com/agentskills/agentskills/tree/main/skills-ref) 参考库来校验你的 skills：

```bash
skills-ref validate ./my-skill
```

这会检查你的 `SKILL.md` 的 frontmatter 是否有效，**并且是否符合所有命名规范**。
