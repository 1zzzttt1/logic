理解代理式循环、内置工具，以及 Claude Code 如何与你的项目交互。

Claude Code 是一个运行在你的终端中的代理式助手。虽然它在编码方面表现出色，但它也可以帮助你完成任何能够通过命令行完成的事情：编写文档、运行构建、搜索文件、研究主题等等。

这篇文章是介绍其**核心架构**、**内置能力**，以及**高效使用它的技巧**。

# 1. 代理式循坏（Agentic loop）

当你给 Claude 一个任务时，它会经历**三个阶段**：**收集上下文、采取行动、验证结果。**

这三个阶段并不是完全割裂的，而是会彼此交织进行。Claude 会在整个过程中持续使用工具，无论是通过搜索文件来理解你的代码、通过编辑来进行修改，还是通过运行测试来检查自己的工作结果。

![图1](./images/knowledge/claude-code/1-1.png)

这个循环会根据你的请求进行调整。

如果你只是询问代码库中的一个问题，可能只需要进行上下文收集。
如果是修复一个 bug，Claude 可能会在这三个阶段之间反复循环多次。
如果是进行一次重构，那么验证结果这一步往往会更深入、更复杂。

Claude 会根据上一步中获取到的信息，判断下一步具体需要做什么，并将几十个动作串联起来，在执行过程中不断修正方向。

**你也是这个循环中的一部分**。**你可以在任何时候打断 Claude**，引导它转向另一个方向、补充更多上下文，或者让它换一种方法尝试。Claude 会自主地推进任务，但同时也会始终对你的输入保持响应。

> 代理式循环由**两个部分**驱动：**负责推理的模型**（只会思考的大语言模型） 和 **负责执行的工具**。Claude Code 充当的是 Claude 外层的代理式运行框架。它提供了工具、上下文管理和执行环境，**把一个语言模型真正变成了一个有能力完成编码任务的智能代理。**

Claude Code 像一具会干活的身体，负责查看项目、修改文件、运行命令；
Claude 这样的⼤语言模型像会思考的脑袋，负责理解任务、分析问题和决定下一步怎么做。

> **claude code 和 claude 是两个不同的东西**。claude是大语言模型（会思考的脑子），claude code 是一个会干活的身体，claude 和 claude code 两个结合，才能形成会思考，又会干活的智能体。

两者结合起来，就形成了一个既会思考、又会动手执行的智能体。
与此同时，Claude Code 对接的大语言模型并不一定只能是 Claude；**如果某个模型服务做了兼容适配**，那么也可以接入其他模型，**比如Minimax官方把自己的模型接口伪装成claude，那么就可以将Claude Code中的大语言模型更换成 Minimax的**

## 1. 模型

​		Claude Code 会**使用 Claude 模型来理解你的代码**，并对任务进行分析和判断。Claude 可以阅读任意编程语言的代码，理解各个组件之间是如何相互关联的，并判断为了实现你的目标，需要修改哪些内容。
​		面对复杂任务时，它还会先把工作拆分成多个步骤，逐步执行，并根据执行过程中获取到的新信息不断调整自己的做法。

​	claude 官方有多种大语言模型可以选择，它们各自有不同的取舍。（如果底层的大语言模型更换了非官方的，比如Minimax，可以对此忽略，不过也可以看下作为了解，希望有一天你能试试官方的，真的很强！）

* **Sonnet** 系列模型 比较适合**处理大多数编码任务**。
* **Opus** 系列模型 在面对**复杂的架构决策**时，推理能力会更强。

你可以在会话中使用 **`/model`** 来切换模型，
也可以在启动时通过 **`claude --model <name>`** 来指定要使用的模型。

当本指南提到“Claude 会选择”或“Claude 会决定”时，指的是**大语言模型本身在进行思考和判断**。

## 2. 工具

工具正是让 Claude Code 具备**“能主动做事”**能力的关键。

* 读取你的代码
* 修改文件
* 运行命令
* 搜索网页
* 与外部服务交互

如果没有工具，Claude 只能输出文字回答。但有了工具之后，Claude 就可以真正动手做事，比如：

每次使用工具后，都会返回新的信息，这些信息又会继续反馈给 Claude，帮助它判断下一步该怎么做。

![图2](./images/knowledge/claude-code/1-2.png)

这些是最主要的能力。除此之外，Claude 还拥有一些工具，用于启动子代理、向你提问，以及处理其他编排类任务。查看完整的工具列表： [Tools available to Claude](https://code.claude.com/docs/en/tools-reference) 

Claude 会根据你的提示词，以及它在执行过程中逐步获取到的信息，来决定该使用哪些工具。

当你说“修复失败的测试”时，Claude 可能会：

1. 运行测试套件，看看是哪些地方失败了
2. 阅读报错输出信息
3. 搜索相关的源代码文件
4. 阅读这些文件，理解代码逻辑
5. 编辑这些文件，修复问题
6. 再次运行测试，确认是否已经修复成功

**每一次使用工具，都会给 Claude 带来新的信息，而这些新信息又会帮助它判断下一步该怎么做。**
这就是“代理式循环”真正运作起来时的样子。



扩展基础能力：内置工具只是基础。你还可以通过以下方式进一步扩展 Claude 的能力：

* 用  [skills](https://code.claude.com/docs/en/skills), 增加它“会做什么”
* 用 [MCP](https://code.claude.com/docs/en/mcp) 连接外部服务
* 用 [hooks](https://code.claude.com/docs/en/hooks) 自动化工作流程
* 用  [subagents](https://code.claude.com/docs/en/sub-agents) 分担任务

这些扩展能力可以理解为搭建在核心“代理式循环”之上的一层增强功能。

如果你想知道该怎么根据自己的需求选择合适的扩展方式，可以查看  [Extend Claude Code](https://code.claude.com/docs/en/features-overview) 



# 2. claude 可以访问什么

本指南主要聚焦于终端环境（cmd、PowerShell）。Claude Code 也可以运行在 VS Code、JetBrains 系列 IDE 以及其他环境中。

当你在某个目录中运行 `claude` 时，Claude Code 就会获得对以下内容的访问能力：

* **你的项目**。包括当前目录及其子目录中的文件；如果你授权，它也可以访问其他位置的文件。
* **你的终端**。凡是你能在命令行里运行的命令，它通常也可以运行，比如构建工具、Git、包管理器、系统工具和脚本。也就是说，只要你能通过命令行完成的事，Claude 通常也能做。
* **你的 Git 状态**。包括当前所在分支、尚未提交的改动，以及最近的提交历史。
* **你的  [CLAUDE.md](https://code.claude.com/docs/en/memory).**。这是一个 Markdown 文件，你可以在里面写项目专属的说明、开发约定，以及希望 Claude **每次进入会话时都知道的上下文信息**。
* **[Auto memory](https://code.claude.com/docs/en/memory#auto-memory)（自动记忆）**。这是 Claude 在你工作过程中**自动保存的一些经验信息**，比如项目中的常见模式、你的个人偏好等。每次会话开始时，它会加载 `MEMORY.md` 的前 200 行或前 25KB 内容，以先到达的限制为准。
* **你配置的扩展功能**。用于连接外部服务的 **MCP 服务器**、用于工作流的 **skills**、用于分担任务的 **subagents**、以及用于浏览器交互的 **Claude in Chrome**

因为 Claude 能看到你的整个项目，所以它可以在整个项目范围内协同工作。

当你让 Claude “修复认证相关的 bug” 时，它可能会先去搜索相关文件，再阅读多个文件来理解上下文，然后在这些文件之间进行联动修改，接着运行测试来验证修复是否成功；如果你要求，它还可以帮你提交这些更改。

这和那种只能看到当前文件的内联代码助手不一样。

# 4. 使用会话进行工作



Claude Code 会在你工作时，**把你的对话内容保存在本地。**

你发送的每一条消息、每一次工具调用，以及每一次执行结果，都会被记录下来。正因为如此，你才可以对会话进行**回退、继续执行，或者分叉出新的会话**。（[rewinding](https://code.claude.com/docs/en/how-claude-code-works#undo-changes-with-checkpoints), [resuming, and forking](https://code.claude.com/docs/en/how-claude-code-works#resume-or-fork-sessions)）

另外，在 Claude 修改代码之前，它还会先为将要变动的文件创建快照，这样如果之后有需要，你就可以把这些改动再恢复回去。

**会话彼此之间是独立的**。每次开启一个新会话时，都会从一个全新的上下文窗口开始，不会自动带上之前会话中的聊天记录。不过，Claude 可以通过**自动记忆（auto memory）在不同会话之间保留一些经验信息**；你也可以把希望**长期生效的说明写进 `CLAUDE.md`**，**让 Claude 在之后的会话中持续参考**。



##  1. 跨分支工作

每一次 Claude Code 对话，都会形成一个与**当前目录**绑定的会话。当你之后继续这个会话时，你只能看到**这个目录下**的那些会话。

Claude 看到的是**你当前分支里的文件内容**。当你切换到其他分支后，Claude 看到的也会变成**新分支中的文件**，但你的**对话历史不会因此消失**，仍然会保留下来。也就是说，**即使你切换了分支，Claude 依然记得你们之前聊过什么。**

由于会话是和目录绑定的，所以你可以通过使用 **[git worktrees](https://code.claude.com/docs/en/common-workflows#run-parallel-claude-code-sessions-with-git-worktrees)** 来同时运行多个 Claude 会话。因为 **git worktree** 可以为不同分支创建各自独立的目录。



## 2. 恢复会话或分叉会话

当你使用 `claude --continue` 或 `claude --resume` 来恢复一个会话时，你会基于**同一个会话 ID**，**从上次停下来的地方继续。**

* `claude --continue`：基于上次对话停下来的地方继续
* `claude --resume`：会让你挑历史会话，选择某一次对话继续

新的消息会直接追加到原来的对话中。

你的完整对话历史会被恢复出来，但**只在当前会话中生效的权限设置不会自动恢复**(聊天记录会保留，但你之前“点过同意”的那些操作许可，不会一起保留。)，**这些权限需要你重新授权**。

![图3](./images/knowledge/claude-code/1-4.svg)

如果你想从**当前会话中分出一条新的分支**，尝试另一种做法，同时又不影响原来的会话，可以使用 `--fork-session` 参数：

```bash
claude --continue --fork-session
```

这会创建一个**新的会话 ID**，同时保留到当前为止的对话历史。原来的会话不会发生变化，仍然保持不变。

和恢复会话（resume）一样，**分叉出来的新会话也不会继承只在原会话中生效的那些权限**，这些权限仍然需要你重新授权。

同一个会话在多个终端中使用：如果你在多个终端里恢复了同一个会话，那么这几个终端都会同时写入同一个会话文件。**来自两个终端的消息会交错在一起**，就像两个人同时在同一本笔记本上写字一样。虽然文件不会损坏，**但整个对话会变得比较混乱。**

**在会话进行的过程中，每个终端通常只能看到自己发出的那些消息；但如果你之后再次恢复这个会话，就会看到所有内容都混在一起、交错出现。**

如果你想基于同一个起点并行开展工作，最好使用 `--fork-session`（将会话分叉开展），这样每个终端都会拥有自己独立、干净的会话。



## 3. 上下文窗口

Claude 的上下文窗口中会包含很多内容，比如：

- 你们的对话历史
- 文件内容
- 命令输出结果
- `CLAUDE.md`
- 自动记忆（Auto memory）
- 已加载的 skills
- 系统指令

随着你不断工作，这些内容会越来越多，上下文空间也会逐渐被占满。

Claude 会自动对上下文进行压缩，但这样一来，**对话前面较早出现的一些指令，可能会慢慢被淡化甚至丢失**。

所以，那些你希望长期一直生效的规则，最好写进 **`CLAUDE.md`** 里。

如果你想查看现在是什么内容在占用上下文空间，可以运行：

```BASH
/context
```

如果你想通过一种可交互的方式了解**有哪些内容会被加载，以及它们是在什么时候被加载的**，
可以查看 [Explore the context window](https://code.claude.com/docs/en/context-window).

### 1. 当上下文窗口被占满时

当你逐渐接近上下文上限时，**Claude Code 会自动帮你管理上下文内容。**

它通常会**先清理较早的工具输出**；如果空间还是不够，就会进一步把**整段对话压缩成摘要**。

你的请求和关键代码片段一般会尽量保留下来，**但对话前面那些比较详细的说明，可能会慢慢丢失**。所以，那些你希望一直生效的长期规则，最好写进 **`CLAUDE.md`**，而不要只依赖聊天记录来保存。

如果你想控制在上下文压缩时**哪些内容应该被保留下来**，可以在 **`CLAUDE.md`** 中添加一个 **“Compact Instructions”** 部分，或者在运行 `/compact` 时额外说明重点内容，比如：

```baSH
/compact focus on the API changes
```

意思就是：**压缩上下文时，请重点保留和 API 改动相关的内容。**

运行 `/context` 可以查看哪些内容在占用上下文空间。MCP 的工具定义默认是延迟加载、按需加载的，所以在 Claude **真正使用某个具体工具之前，占用上下文的只有工具名称**。**运行 `/mcp` 可以查看每个服务器分别消耗了多少上下文。**

### 2. 用 skills 和 subagents 来管理上下文。

除了压缩上下文之外，你还可以使用其他功能来控制哪些内容会被加载进上下文。

**skills 是按需加载的**。Claude 会在会话开始时**看到技能描述**，但**只有在实际使用某个技能时，才会加载它的完整内容**。对于你手动调用的技能，可以设置 `disable-model-invocation: true`，这样在你真正需要之前，这些描述就不会进入上下文。

[Subagents](https://code.claude.com/docs/en/sub-agents) 子代理会**获得一份全新的独立上下文**，**和你的主对话完全分开**。它们执行任务时，不会让你的主上下文变得臃肿。完成后，它们**只会返回一份总结**。正因为这种隔离机制，子代理在长会话里特别有用。

查看 **[context costs](https://code.claude.com/docs/en/features-overview#understand-context-costs)** 了解每个功能会占用多少上下文成本，查看  [reduce token usage](https://code.claude.com/docs/en/costs#reduce-token-usage) 获取管理上下文的节省技巧。

# 5. 通过检查点和权限机制来确保安全。

Claude 有两种安全机制：**检查点** 可以让你撤销文件更改，**权限机制** 则用来控制 Claude 在不询问你的情况下可以执行哪些操作。

## 1. 使用检查点撤销更改。

每一次文件编辑都是可撤销的。在 Claude 修改任何文件之前，它都会先对当前内容创建一个快照。要是出了问题，你可以连按两次 `Esc` 回退到之前的状态，或者直接让 Claude 帮你撤销。

检查点是**当前会话本地**的机制，和 Git 是分开的。它们只覆盖**文件改动**。那些会影响远程系统的操作（比如数据库、API、部署）无法被检查点记录，这也是为什么 Claude 在运行具有外部副作用的命令前会先征求你的确认。

意味着，如果回退检查点，**它不能撤销“已经影响外部世界”的操作**，比如这些就不行：

* 执行 SQL，把数据库数据删了
* 调接口，把线上数据改了
* 发起部署，把新版本推到服务器

检查点只能恢复**你本地文件**，但没法让外部系统“时光倒流”。

## 2. 控制 Claude 可以执行的操作。

按 `Shift + Tab` 可以循环切换不同的权限模式：

* **默认模式（Default）**：Claude 在编辑文件和执行 shell 命令前都会先询问你。
* **自动接受编辑（Auto-accept edits）**：Claude 修改文件时不再询问，但执行命令时仍会先询问。
* **计划模式（Plan mode）**：Claude 只使用只读工具，先生成一个计划，等你批准后再执行。
* **自动模式（Auto mode）**：Claude 会结合后台安全检查来评估并执行所有操作。**目前这是一个研究预览功能。**

你也可以在 `.claude/settings.json` 里允许特定命令，这样 Claude 就不必每次都询问你了。这对一些你信任的命令很有用，比如 `npm test` 或 `git status`。这些设置的作用范围可以从组织级策略一直细化到个人偏好设置。详情请查看  [Permissions](https://code.claude.com/docs/en/permissions) 。



# 6. 高效使用 Claude Code。

这些技巧可以帮助你更高效地使用 Claude Code，并获得更好的结果。



## 1. 向 Claude Code 寻求帮助。

Claude Code 也可以教你如何使用它。你可以问这样的问题，比如“我该怎么设置 hooks？”或者“我的 `CLAUDE.md` 最佳结构应该怎么写？”，Claude 都会为你解释。

内置命令也会引导你完成设置：

* `/init` 会一步步带你为项目创建 `CLAUDE.md`
* `/agents` 会帮助你配置自定义子代理
* `/doctor` 会诊断你安装过程中常见的问题



## 2. 这是一场对话。

Claude Code 是对话式的。你不需要一开始就写出完美的提示词。先说出你想要什么，再逐步细化：

```txt
Fix the login bug
```

【Claude 进行调查，并尝试某些方法】

```txt
这还不太对。问题出在会话处理这部分。
```

【Claude 调整处理思路】

**当第一次尝试不够正确时，你不需要从头再来，而是继续迭代优化。**



### 1. 中途打断并引导方向。

你可以在任何时候打断 Claude。如果它走偏了，只要输入你的纠正内容并按下 Enter。Claude 就会停止当前正在做的事，并根据你的输入调整处理方式。你不需要等它做完，也不需要重新开始。



## 3. 一开始就尽量说具体一些。

你最开始给出的提示越精确，后续需要纠正的次数就越少。可以明确指出具体文件、说明限制条件，并给出可参考的示例模式。

```txt
持卡信息已过期的用户在结账流程中会遇到问题。
请检查 src/payments/ 里的相关问题，尤其关注 token 刷新这部分。
先写一个会失败的测试，然后再修复它。
```

模糊的提示也能用，但你会花更多时间去纠正和引导。像上面那样具体的提示，往往第一次尝试就更容易成功。



## 4. 给 Claude 一个可以用来验证的依据。

当 Claude 能自行检查自己的工作结果时，表现会更好。你可以提供测试用例、贴上预期 UI 的截图，或者明确说明你想要的输出结果。

```txt
实现 validateEmail。
测试用例：'user@example.com' → true，'invalid' → false，'user@.com' → false。
完成后再运行测试。
```

可以把它记成一个简单公式：**任务目标 + 验收标准 + 完成后自检**

对于视觉类工作，可以贴一张设计稿截图，并让 Claude 将自己的实现结果与这张截图进行对比。

> 你在让 Claude Code 干活时，**不要只说“帮我实现一下”**，而要顺手给它一个**可验证的标准**。
>  这样它做完以后，不是“自我感觉良好”，而是能拿你给的标准去检查自己到底做对没有。



## 5. 先探索，再实现。

对于复杂问题，先把**调研**和**写代码**分开。可以使用 **计划模式**（按两次 `Shift + Tab`）先分析代码库：

```txt
阅读 src/auth/，先理解我们是如何处理会话的。
然后再制定一个为项目添加 OAuth 支持的计划。
```

**先审查这个计划，通过对话不断完善它**，然后再让 Claude 去实现。相比一开始就直接写代码，这种“两阶段”方式通常能得到更好的结果。



## 6. 多说“我要什么结果”，少说“你必须一步一步按我这个细节流程做”。

* **不太好的方式：过度指挥**

  > 打开 `src/auth/session.ts`，看第 20 行，把 `token` 改成 `accessToken`，然后去 `user.ts` 加一个字段，再去 `api.ts` 改请求头。

  这种方式的问题是：

  * 你自己已经承担了大部分分析工作
  * 一旦你判断错了，Claude 也会沿着错方向走
  * 它只能“照做”，很难帮你发现更深层的问题

* **更好的方式：有效委托**

  > 登录状态经常失效。请先阅读 `src/auth/` 相关代码，找出 session 刷新链路的问题，给我一个简短计划。确认后再实现，并补上失败测试。

  这种方式更好，因为你给了：

  * 目标：修复登录状态失效
  * 范围：`src/auth/`
  * 流程：先分析，再计划，再实现
  * 验收：补失败测试

  Claude 就能自己做更完整的工作。

你不需要明确指定要读取哪些文件，或者要运行哪些命令。Claude 会自己判断这些。
